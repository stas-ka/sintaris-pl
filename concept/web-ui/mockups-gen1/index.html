Telegram Adaptation Notes

🔄 Same intent model, different renderer

The backend intent parser is identical. Only the rendering layer changes — web shows visual cards and animations, Telegram renders text + inline keyboards.

🎤 Voice is native in both channels

Telegram voice messages → STT → same intent routing as the web orb. The flow is nearly identical. No typing required in either channel.

⚡ Inline keyboards = What's Next chips

The contextual chips on web become Telegram inline keyboard buttons. Same LLM-generated suggestions, same logic.

🔗 Deep links: Telegram → Web

An "Open in web" button in Telegram links directly to the note/event in the web UI for rich editing. Best of both worlds.

Architecture principle: One intent API → two renderers. Adding WhatsApp, Discord, or any 3rd channel requires only a new renderer (~100 lines). The intent logic, LLM routing, and data layer are shared.
What you see
The Hub — the new landing page. A breathing orb is the primary call to action. Recent activity shows the 3 most recent items across all modules. Quick chips offer one-tap shortcuts to common intents. The sidebar is icon-only and secondary.
UX Principles
P1 — Voice First
P2 — Intent Over Navigation
The user never navigates to "Notes" to create a note. They speak from the Hub and the system routes itself. Navigation becomes invisible.
Why it matters
Current problem: users must know where functions live. With the Hub, they only need to express what they want. Dramatically reduces cognitive load. Makes the app functional on voice-only or low-vision devices.
What you see
Voice recording is active. The orb expands with vivid rings. An animated waveform shows audio level. The live transcript streams onto the screen as the user speaks — confirming they are being heard in real time.
UX Principle
P1 — Voice First
Live transcription is a confidence builder. When users see their words appear as they speak, they trust the system and are more comfortable with longer, more natural commands.
Technical note
Uses existing MediaRecorder API. Live transcript uses Web Speech API as a display layer while full Vosk STT runs server-side. The server result corrects the live preview on completion.
What you see
The Intent Card slides up from the bottom. It shows the parsed action type, extracted entities (title, TOC setting, mode), and a confidence score. A progress bar counts down 2 seconds to auto-proceed.
UX Principle
P3 — Transparent Understanding
The user sees exactly what was understood before the action runs. If the title was misheard, they tap "Correct title" and fix only that part — no need to repeat the full command.
Why 2-second auto-proceed?
Long enough to read and correct. Short enough to feel instant for experienced users. The countdown bar is visible and cancellable. Confident users never have to tap anything — happy-path is frictionless.
What you see
The Note Editor opened automatically with the title pre-filled from the voice command. The "🎤 Listening" badge in the topbar shows voice dictation is still active. The TOC panel is being built live as sections are detected in the dictated content.
UX Principles
P2 — Intent Over Navigation
P5 — Contextual Continuity
The app transitioned from Hub to Notes Editor without any navigation tap. Voice continues seamlessly. The floating orb (bottom right) stays available for the next command at all times.
TOC auto-generation
The assistant detects section cues in natural speech ("Section one:", "Next, the bread recipe") and populates the TOC panel live. The user never writes a table of contents — they just speak naturally and structure emerges.
What you see
A Confirmation Card overlays the blurred editor. It shows a preview of the note: title, section structure, word count, metadata. Three action buttons: Save, Edit (return to editor), Cancel.
UX Principle
P4 — Confirmation Without Friction
The card is informative, not just a yes/no prompt. The user sees exactly what will be saved without needing to scroll the editor. One tap to save — no additional friction. Edit returns to the editor seamlessly.
Confirm / skip decision rule
Always confirm: Save note, create event, send email, delete anything.

Never confirm: Open a page, read content, show a list, search.

Rule: if it can't be easily undone, confirm first.
What you see

UX Principles
P5 — Contextual Continuity
P6 — One Thread, All Modes
The chips are not static tabs — they are context-aware. After a cookbook note, the system offers: chat about it, read aloud, share by email, add a cooking event. These five chips are specific to this action, not generic navigation.
Chip lifecycle
The "What's Next" chips auto-disappear after 15 seconds if not tapped. After that, the Hub returns to idle and the orb breathes again, inviting the next interaction. Nothing is left hanging.
What you see
Chat mode activated by the user's voice command "I want to chat with you". The previous note action appears as a compact action card at the top of the thread — a persistent record. The conversation continues naturally below it.
UX Principle
P6 — One Thread, All Modes
The thread is the memory of the session. Switching to chat doesn't reset context — the assistant knows exactly which note was just created and can discuss it. The session is continuous even when the mode changes.
Action cards in thread
Every completed action (note saved, event added, mail sent) leaves a card in the thread with quick-action buttons (Open / Edit / Read aloud). Users can act on previous items without leaving the conversation. The thread becomes a unified activity log.
What you see
The identical note-creation flow rendered in Telegram. Voice message → bot parses intent → inline keyboard for confirmation → content dictation → save confirmation → "What's Next" keyboard. Same logic, adapted presentation.
Architecture Principle
One Intent API → Many Renderers
Intent parsing and routing live in one backend layer. Web and Telegram are renderers. Adding WhatsApp or Discord requires only a new renderer (~100 lines), not new logic. A "Deep Link" button opens any Telegram item in the web UI for rich editing.
Telegram advantages

No login needed: already authenticated.
Voice native: voice messages feel natural in Telegram.
