## 1. Общие сведения

**Наименование системы:**  
Лёгкая CRM-система управления контактами клиентов и планирования follow-up взаимодействий.

**Основание для разработки:**  
Рост клиентской базы и невозможность поддерживать совместную работу в текущем инструментарии (Excel / Google Таблицы).

**Заказчик:**  
Внешний. Внедрять будет продвинутый IT-специалист, опыт работы с Linux, nginx, MySQL, серверной инфраструктурой. Опыт настройки CRM данного класса — отсутствует.

**Ответственный за приёмку:**  
Администратор системы.

## 2. Назначение и цели системы

### 2.1 Назначение

Система предназначена для:
- ведения единой базы данных клиентов с полной контактной информацией;
- сохранения и структурирования контактов;
- фиксации истории всех коммуникаций и касаний с каждым клиентом;
- планирования и контроля повторных обращений и касаний (follow-up);
- планирования мероприятий и отслеживания участия клиентов в них;
- подбора клиентов под конкретный оффер, мероприятие или тему;
- организации совместной работы нескольких сотрудников с общей базой данных;
- автоматической отправки уведомлений о задачах и напоминаниях;
- предоставления удобного интерфейса для ежедневной работы.

### 2.2 Цели внедрения

| # | Цель | Измеримый результат |
|---|---|---|
| 1 | Перейти от Excel к полноценной CRM | 100% клиентов ведётся в новой системе |
| 2 | Исключить потерю follow-up | 0 пропущенных задач при наличии напоминаний |
| 3 | Совместная работа без конфликтов | До 5 пользователей одновременно в единой базе |
| 4 | Хранение данных на серверах в России | Соответствие требованиям ФЗ-152 |
| 5 | Рассылка уведомлений и приглашений клиентам | Доставка приглашений на мероприятия и напоминаний через email или мессенджер |

### 2.3 Решаемые проблемы

Текущие инструменты (Excel / Google Таблицы) не обеспечивают:

- управление растущей клиентской базой (ожидаемый объём: до 50 000 записей);
- совместную работу нескольких сотрудников без конфликтов версий;
- планирование follow-up с привязкой к клиенту и ответственному сотруднику;
- хранение полной истории коммуникаций с каждым клиентом.

## 5. Функциональные требования

### 5.1 Управление клиентами (карточка клиента)

Система должна хранить и отображать следующие данные о каждом клиенте.

#### 5.1.1 Основные контактные данные

| Поле | Тип | Обязательность |
|---|---|---|
| Фамилия | Текст | Обязательно |
| Имя | Текст | Обязательно |
| Отчество | Текст | Необязательно |
| Телефон (основной) | Телефон | Обязательно |
| Email | Email | Необязательно |

#### 5.1.2 Дополнительные контакты

| Поле | Тип | Обязательность |
|---|---|---|
| Telegram (ник или ссылка) | Текст | Необязательно |
| Instagram | Текст | Необязательно |
| ВКонтакте | Текст | Необязательно |
| Другие мессенджеры / соцсети | Текст (свободное поле) | Необязательно |

#### 5.1.3 Служебные поля

| Поле | Описание |
|---|---|
| Источник лида | Из справочника (см. 5.2) с полем комментария |
| Статус клиента | Активен / В работе / Архив / Неактуален |
| Теги | Список меток; проставляются вручную или автоматически через AI |
| Краткое описание (summary) | Свободный текст; заполняется вручную или генерируется AI |
| Сегмент | Из справочника; определяется вручную или через AI |
| Дата создания | Автоматически |
| Кем создан | Автоматически (текущий пользователь) |
| Ответственный менеджер | Из списка пользователей |

#### 5.1.4 Требования к операциям

- **Создание** клиента с заполнением обязательных полей.
- **Просмотр** карточки клиента: все данные, история, задачи на одной странице.
- **Редактирование** любого поля карточки.
- **Поиск** — по всем полям карточки клиента (подробнее в разделе 5.6).
- **Обнаружение дублей** — система предупреждает о возможном дубле при совпадении телефона или email.
- **Архивирование** — смена статуса клиента на «Архив» или «Неактуален»; архивные записи сохраняются, но по умолчанию скрыты из рабочих списков.
- **Удаление** — только для роли Администратор, с запросом подтверждения.

---

### 5.2 Источник лида

Каждый клиент должен содержать поле «Источник лида».

#### 5.2.1 Предустановленные значения справочника

- Рекомендация
- Реклама
- Холодный звонок
- Личный контакт
- Instagram
- ВКонтакте / VK Макс
- Telegram
- Сайт / форма
- Другое

#### 5.2.2 Требования к справочнику

- Администратор может добавлять, редактировать и скрывать значения справочника.
- К каждому значению источника можно добавить текстовый комментарий (уточнение).
- Поле обязательно при создании клиента.

---

### 5.3 История коммуникаций

На карточке каждого клиента ведётся хронологический журнал всех взаимодействий.

#### 5.3.1 Данные записи в истории

| Поле | Описание |
|---|---|
| Дата и время | Автоматически (с возможностью ручного ввода ретроспективно) |
| Тип коммуникации | Звонок / Встреча / Email / Telegram / Заметка |
| Содержание | Краткое описание разговора: потребности клиента, суть беседы |
| Результат | Итог взаимодействия (договорились, отказ, думает и пр.) |
| Автор записи | Текущий пользователь (автоматически) |

#### 5.3.2 Пример записи

```
18.03.2026 | Звонок
— Обсудили услугу X. Клиент заинтересован.
— Потребность: хочет уточнить цену и сроки.
— Результат: договорились отправить КП, перезвонить 21.03.
Автор: Иванова А.
```

#### 5.3.3 Требования

- Все записи отображаются в хронологическом порядке (новые — сверху).
- Запись нельзя удалить (только администратор).
- При добавлении записи в историю — предлагать создать follow-up задачу.

---

### 5.4 Задачи и планирование follow-up

Система должна поддерживать создание задач, привязанных к клиенту.

#### 5.4.1 Поля задачи

| Поле | Описание |
|---|---|
| Клиент | Привязка к карточке клиента |
| Описание задачи | Текстовое поле (например: «Позвонить, уточнить решение») |
| Дата выполнения | Дата + время (обязательно) |
| Ответственный | Из списка пользователей |
| Статус | Активна / Выполнена / Просрочена |
| Приоритет | Низкий / Средний / Высокий |

#### 5.4.2 Требования

- Задача может быть создана из карточки клиента или из общего раздела «Задачи».
- При создании записи в истории — возможность сразу создать follow-up.
- Статус «Просрочена» устанавливается автоматически при наступлении срока без отметки выполнения.
- Ответственным по умолчанию назначается текущий пользователь.
- Задача видна в карточке клиента и в общем списке задач.

---

### 5.5 Напоминания

Система должна уведомлять сотрудников о предстоящих и просроченных задачах.

#### 5.5.1 Типы напоминаний

| Тип | Описание |
|---|---|
| Уведомление в интерфейсе | Значок / счётчик непрочитанных задач на сегодня |
| Список «Задачи на сегодня» | Видимый блок на главном экране (дашборд) |
| Просроченные задачи | Отдельный блок / выделение красным |

---

### 5.6 Поиск, фильтрация и сортировка

В разделе «Клиенты» система должна обеспечивать:

#### 5.6.1 Поиск

- Полнотекстовый поиск по: ФИО, телефону, email, Telegram, Instagram, тегам, сегменту.
- Мгновенный поиск (начиная с 2–3 введённых символов).
- Результаты поиска отображаются в табличном виде.

#### 5.6.2 Фильтры

| Фильтр | Возможные значения |
|---|---|
| Источник лида | Любое значение из справочника |
| Статус клиента | Активен / В работе / Архив / Неактуален |
| Теги / сегмент | Выбор из имеющихся тегов и сегментов |
| Ответственный менеджер | Список пользователей |
| Дата добавления | Диапазон дат |
| Дата следующего follow-up | Диапазон дат |
| Есть просроченные задачи | Да / Нет |
| Есть следующий шаг | Да / Нет |

#### 5.6.3 Сортировка

Сортировка по любому столбцу таблицы (по возрастанию / убыванию):
- Дата добавления
- Имя клиента
- Источник
- Дата следующего follow-up
- Ответственный

---

### 5.7 Управление пользователями (только для Администратора)

- Создание новых учётных записей.
- Редактирование данных пользователей (ФИО, email, Telegram).
- Назначение роли: Администратор / Оператор.
- Блокировка / деактивация учётной записи без удаления данных.
- Сброс пароля пользователя.

---

### 5.8 Справочники (только для Администратора)

- Управление значениями источников лидов (добавление, редактирование, скрытие).
- Управление типами коммуникаций (Звонок, Встреча, Email и пр.).
- Управление статусами задач.
- Управление значениями сегментов клиентов (добавление, редактирование, скрытие).
- Управление предустановленными тегами (добавление, редактирование).

### 5.9 Планирование мероприятий

Система должна обеспечивать создание, хранение и отображение мероприятий, а также планирование участия клиентов в них.

#### 5.9.1 Поля мероприятия

| Поле | Описание | Обязательность |
|---|---|---|
| Название | Краткое наименование мероприятия | Обязательно |
| Тема | Развёрнутое описание темы и содержания | Обязательно |
| Дата и время | Дата, время начала и окончания | Обязательно |
| Формат | Онлайн / Офлайн / Гибридный | Обязательно |
| Место / ссылка | Адрес или ссылка на онлайн-площадку | Необязательно |
| Целевая аудитория | Текстовое описание: кому предназначено мероприятие | Обязательно для AI-подбора |
| Ключевые слова | Список тегов для AI-подбора участников | Необязательно |
| Ответственный | Из списка пользователей | Обязательно |
| Статус | Черновик / Готово к рассылке / Рассылка выполнена / Завершено | Автоматически |

#### 5.9.2 Список участников

Каждое мероприятие имеет связанный список участников из базы клиентов.

| Поле участника | Описание |
|---|---|
| Клиент | Ссылка на карточку клиента |
| Статус приглашения | Не приглашён / Приглашён / Подтвердил / Отказался |
| Оценка AI | Подходит / Не подходит |
| Обоснование AI | Почему рекомендован данный клиент (автоматически) |
| Рекомендуемый формат приглашения | Как лучше пригласить данного клиента (от AI) |

#### 5.9.3 Требования к операциям

- **Создание** мероприятия с заполнением обязательных полей.
- **Редактирование** всех полей мероприятия до его завершения.
- **AI-подбор участников** — автоматическая оценка клиентов базы на соответствие мероприятию (см. сценарий 4, раздел 5.10).
- **Ручное добавление** клиентов в список участников помимо AI-подбора.
- **Утверждение списка** участников перед рассылкой — только после ручного подтверждения ответственным.
- **Запуск рассылки** приглашений по утверждённому списку через email или мессенджер.
- **Регистрация напоминаний** клиентам о предстоящем мероприятии (задачи в CRM или внешние уведомления).
- **Привязка к карточке клиента** — на карточке клиента отображается список мероприятий, к которым он привязан.
- **Просмотр** мероприятия в интегрированном или внешнем календаре.

### 5.10 Сценарии автоматизации с поддержкой AI

Раздел описывает ключевые автоматизированные процессы, реализуемые с помощью системы автоматизации (n8n или аналог) в связке с AI.

#### 5.10.1 Сценарий 1. Добавление нового контакта

- Оператор заполняет карточку нового контакта и сохраняет её.
- Система проверяет заполнение обязательных полей.
- Заметки и информация об интересах клиента передаются в AI.
- AI возвращает: теги, краткое описание (summary), предполагаемый сегмент.
- Результат автоматически записывается в карточку клиента.
- Если указана дата следующего касания — в календаре создаётся соответствующее событие.

#### 5.10.2 Сценарий 2. Постановка следующего касания

- Оператор вносит данные следующего касания: дата/время, тип, краткая заметка, ответственный.
- В интегрированном или внешнем календаре создаётся событие, содержащее:
    - имя и контакты клиента;
    - цель касания;
    - краткую заметку;
    - ссылку на карточку клиента.
- Статус клиента обновляется автоматически.

#### 5.10.3 Сценарий 3. Ежедневный контроль

- Каждое утро система автоматически анализирует базу контактов и отбирает:
    - касания, запланированные на сегодня;
    - просроченные касания (срок прошёл, задача не закрыта);
    - контакты без следующего запланированного шага;
    - новые контакты без записи первого касания.
- Из отобранных данных формируется дневная сводка и отображается в разделе «Задачи дня» на дашборде.
- Сводка автоматически отправляется ответственным сотрудникам на email и/или в мессенджер.

#### 5.10.4 Сценарий 4. AI-подбор контактов под мероприятие

- Мероприятие создаётся в системе с заполнением параметров целевой аудитории. Пример:
    - тема: «AI для малого бизнеса»;
    - целевая аудитория: предприниматели, эксперты, кто хочет автоматизировать рутину;
    - ключевые слова: AI, автоматизация, делегирование, продажи, воронки.
- Системой выполняется предварительная фильтрация контактов — исключаются:
    - клиенты со статусом «Архив» или «Неактуален»;
    - клиенты без контактных данных.
- Отобранные активные контакты передаются в AI вместе с описанием мероприятия.
- AI оценивает каждый контакт и возвращает:
    - вердикт: подходит / не подходит;
    - обоснование: почему подходит, какая потребность совпадает;
    - рекомендацию: какой формат приглашения лучше использовать.
- Результат заносится в раздел «Подборки» на карточке мероприятия.
- Администратор просматривает подборку и утверждает список участников.
- После утверждения:
    - клиенты получают приглашение (email или мессенджер);
    - в CRM регистрируются напоминания о предстоящем мероприятии.

Действия: «Создать мероприятие», «Подобрать участников (AI)», «Отправить приглашения», «Зарегистрировать напоминания».

## 7. Требования к интеграциям

### 7.1 AI (искусственный интеллект) — обязательна поддержка

**Назначение:** Автоматическая обработка данных клиентов и подбор контактов; генерация тегов, summary, сегментов; оценка соответствия клиентов мероприятиям.

**Требования:**
- Система должна поддерживать подключение к внешним AI-сервисам (OpenAI GPT, Yandex GPT и аналоги) — через прямой API или через сервис автоматизации (n8n и аналоги).
- AI вызывается по триггерам из CRM (создание контакта, запуск подбора под мероприятие) и возвращает результат в карточку клиента или в лист подборки.
- Конкретный AI-провайдер не фиксируется — выбирается при реализации.

---

### 7.2 Автоматизация через n8n или аналог

**Назначение:** Оркестрация автоматических сценариев (см. раздел 5.10): ежедневная сводка, уведомления, вызов AI, синхронизация с календарём.

**Требования:**
- Система должна предоставлять API (REST или webhook) для запуска сценариев из внешних систем автоматизации.
- Система должна принимать входящие запросы (webhook) от внешних систем для обновления данных или создания событий.
- Конкретный инструмент автоматизации не фиксируется (n8n, Make, Zapier и аналоги).

---

### 7.3 Календарь

**Назначение:** Создание событий (касания, мероприятия) и отображение расписания.

**Требования:**
- Система должна поддерживать встроенный календарь или интеграцию с внешним (Google Calendar, Яндекс Календарь и аналоги).
- При создании задачи/касания с датой — событие создаётся в календаре автоматически.
- Событие содержит: имя клиента, цель касания, контакты, краткую заметку, ссылку на карточку.

---

### 7.4 Уведомления сотрудников

**Назначение:** Информирование сотрудников о задачах, просрочках, новых назначениях.

**Требования:**
- Уведомления отправляются через email и/или мессенджер (Telegram или аналог).
- Канал доставки не фиксируется — настраивается при реализации.
- Сотрудник может указать предпочтительный канал уведомлений в своём профиле.

---

### 7.5 Уведомления и рассылки клиентам — опционально

**Назначение:** Отправка приглашений на мероприятия, напоминаний клиентам.

**Требования:**
- Система должна поддерживать отправку сообщений клиентам через email и/или мессенджер (Telegram и аналоги).
- Рассылка клиентам выполняется по утверждённому списку после ручного подтверждения.
- Канал связи выбирается исходя из наличия контакта в карточке клиента.

---

### 7.6 IP-телефония — перспектива

**Назначение:** Звонки клиентам из интерфейса CRM с автоматическим созданием записи в истории коммуникаций.

**Требования:**
- Платформа должна поддерживать подключение телефонии без смены CRM-системы.
- Входит в перспективные расширения (раздел 12).

## 9. Требования к безопасности

### 9.1 Аутентификация

- Вход в систему по логину (email) и паролю.
- Пароль: минимум 8 символов, требуется сочетание цифр и букв.
- Сессия: автоматическое завершение после 8 часов неактивности.
- Опционально: двухфакторная аутентификация (2FA).

### 9.2 Разграничение доступа

- Права доступа проверяются на стороне **сервера** для каждого действия.
- Оператор не может получить доступ к данным или функциям за пределами своей роли, в том числе через прямые URL-запросы.
- Удаление данных доступно только Администратору.

### 9.3 Защита данных

- Все соединения только по **HTTPS**.
- Персональные данные клиентов хранятся только на серверах в России (ФЗ-152).
- Пароли хранятся только в виде хэша.

### 9.4 Защита от атак

- Защита от SQL-инъекций, XSS, CSRF.
- Ограничение количества попыток входа: не более 5 за 10 минут, после — временная блокировка.

### 9.5 Журнал действий

- Фиксируются ключевые события: вход/выход, создание/редактирование/удаление клиентов, задач, записей в истории.
- Журнал доступен только Администратору.
- Записи журнала не редактируются и не удаляются.

## 13. Критерии приёмки

Система считается готовой к вводу в эксплуатацию при выполнении всех следующих условий.

### 13.1 Функциональные проверки

| Сценарий | Ожидаемый результат |

| Вход с неверным паролем 5 раз подряд | Временная блокировка аккаунта/IP |
| Создание клиента с обязательными полями | Клиент создан, отображается в списке |
| Попытка создания дубля (тот же телефон) | Система предупреждает о возможном дубле |
| Добавление записи в историю коммуникаций | Запись появляется в карточке хронологически |
| Создание задачи с датой и ответственным | Задача видна в разделе «Задачи» и на дашборде |
| Задача с истёкшим сроком (не выполнена) | Статус «Просрочена», отображается в блоке просроченных |
| Поиск по телефону | Нужный клиент отображается в результатах |
| Создание мероприятия и запуск AI-подбора | Список подходящих клиентов сформирован с оценкой AI |
| Ежедневная сводка | Задачи на сегодня, просрочки, контакты без шага — собраны и доставлены |
| Уведомление сотрудника | Сообщение о задаче доставлено в настроенный канал |
| Оператор обращается к настройкам системы | Доступ запрещён |
| Оператор пытается удалить клиента | Действие недоступно |
| Экспорт базы клиентов | Файл Excel/CSV содержит все данные |
| Резервная копия | Файл создан по расписанию, хранится на отдельном хранилище |

### 13.2 Нефункциональные проверки

| Параметр | Требование |
|---|---|
| Загрузка страницы списка клиентов | ≤ 2 сек при 1000+ записях |

| Мониторинг | При недоступности сервиса — уведомление администратора |

### 13.3 Пользовательская приёмка

- Пользователи всех ролей прошли демонстрацию системы.
- Краткая инструкция для операторов подготовлена.
- Все операторы успешно выполнили: добавление клиента, запись в историю, создание задачи, получение уведомления.
