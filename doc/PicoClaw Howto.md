# PicoClaw Howto

## Что важно понять сразу

PicoClaw в этом проекте расширяется гибридно:

1. Ядро `picoclaw` (Go-бинарник) отвечает за LLM-агента, каналы и часть встроенных инструментов.
2. Кастомная функциональность добавляется внешними сервисами и скриптами (в первую очередь Python + systemd), которые вызывают `picoclaw` как subprocess (`picoclaw agent -m ...`).
3. Оркестрация и включение/выключение возможностей делаются через `~/.picoclaw/config.json`, `bot.env`, systemd unit-файлы и deploy/setup shell-скрипты.

Итого: это не только "пишем Python", и не только "чисто внутренний фреймворк". Это связка: **core picoclaw + внешние сервисы + конфигурация + сервис-менеджмент**.

## Техническая архитектура расширений

### 1) Core слой (picoclaw)

- Бинарник: `/usr/bin/picoclaw`
- Основной конфиг: `~/.picoclaw/config.json`
- Что управляется в конфиге:
  - `model_list`, `agents.defaults`
  - `channels` (telegram/discord/slack/...)
  - `tools` (web/exec/cron/skills/...)

То есть базовые "способности" и провайдеры задаются на уровне ядра.

### 2) Integration слой (проектные сервисы)

В этом репозитории добавлены отдельные сервисы:

- `voice_assistant.py` — оффлайн STT/TTS + hotword, затем вызов `picoclaw agent`.
- `telegram_menu_bot.py` — отдельный UX-бот с меню, режимами, обработкой voice notes.
- `gmail_digest.py` — отдельный агент digest по расписанию.

Эти модули не встраиваются в бинарник picoclaw, а живут рядом и общаются с ним через CLI.

### 3) Runtime слой

- systemd юниты (`picoclaw-gateway`, `picoclaw-voice`, `picoclaw-telegram`)
- env-переменные (`BOT_TOKEN`, `ALLOWED_USERS`, `PICOCLAW_BIN`, `PIPER_MODEL`, ...)
- cron (для задач по расписанию)

Расширение считается "правильно подключенным", когда у него есть:
- отдельный скрипт,
- unit-файл,
- конфиг/env,
- проверка логов и тестовый сценарий.

## Как это происходит на практике

Когда пользователь просит новую функцию (пример: "добавь новый канал/новый режим/новую автоматизацию"), типовой pipeline такой:

1. Добавляется/меняется конфиг `~/.picoclaw/config.json` (модель, канал, tool, лимиты).
2. Пишется скрипт (обычно Python), который реализует новую логику.
3. Скрипт вызывает `picoclaw agent -m` для LLM-части и дополняет это своей бизнес-логикой.
4. Добавляется systemd unit + deploy script.
5. Настраиваются переменные в `bot.env`/`.pico_env`.
6. Делается smoke-test и проверка логов.

## Ответ на ваш вопрос "Python-код или специальный framework?"

Коротко: **оба варианта, но в разных слоях**.

- Если нужно расширить поведение конкретного решения (как в этом репо) — чаще пишется Python/шеловые скрипты и запускается как сервис.
- Если нужно расширить базовые возможности ядра (каналы/tools/skills на уровне самого picoclaw) — это делается через его конфиг и поддерживаемые встроенные механизмы.

На этом проекте основной путь расширения: **внешние сервисы вокруг ядра picoclaw**.

## Рекомендуемые правила расширения (best practices)

1. Держите `picoclaw` как отдельный "LLM backend", а прикладную логику — в отдельных сервисах.
2. Не хардкодьте секреты в коде: только env (`bot.env`, `.pico_env`), права `chmod 600`.
3. Для каждой новой фичи создавайте отдельный unit-файл, а не "монолит" в одном процессе.
4. Всегда задавайте timeout для subprocess-вызовов (`picoclaw`, `ffmpeg`, `piper`).
5. Добавляйте graceful fallback (ошибка STT/TTS/LLM не должна ронять весь сервис).
6. Логи в отдельные файлы на каждый сервис (`voice.log`, `telegram_bot.log`, `digest.log`).
7. Делайте idempotent setup/deploy-скрипты (повторный запуск не ломает систему).
8. Держите конфиг-слой отделенным от кода: модель/каналы/пути/лимиты — в env или `config.json`.
9. Перед запуском проверяйте зависимости и файлы моделей (`vosk`, `piper`, модели).
10. Для опасных операций (System Chat/exec) добавляйте подтверждение пользователя и allowlist/denylist.

## Шаблон добавления новой функции

1. Определите trigger:
   - голос,
   - Telegram callback/команда,
   - cron,
   - внешний webhook.
2. Реализуйте модуль в `src/` (например, `new_feature.py`).
3. Если нужна LLM-логика, вызывайте `picoclaw agent -m`.
4. Добавьте env-конфигурацию в `bot.env.example`/документацию.
5. Создайте systemd unit в `src/services/`.
6. Добавьте deploy/setup script в `src/setup/`.
7. Протестируйте:
   - ручной запуск скрипта,
   - запуск через systemd,
   - рестарт/падение/восстановление,
   - логи.

## Когда менять ядро picoclaw, а когда писать внешний модуль

Меняйте `config.json` ядра, если:
- нужно включить/выключить канал,
- сменить модель/провайдера,
- поменять встроенные tool-настройки.

Пишите внешний модуль, если:
- нужна специфичная бизнес-логика проекта,
- нужна интеграция с локальным железом (микрофон/аудио),
- нужен кастомный UX (меню-бот, режимы, цепочки действий),
- нужна сложная pre/post-обработка до/после LLM.

## Минимальный чеклист "сделано правильно"

- Функция выделена в отдельный файл и сервис.
- Все секреты вынесены в env.
- Есть timeouts и обработка ошибок.
- Есть unit/service + автостарт.
- Есть краткая инструкция в README/doc.
- Есть smoke test с ожидаемым результатом.

## Может ли PicoClaw сам себя расширять по просьбе пользователя

Короткий ответ: частично да, полностью автоматически - обычно нет.

Что PicoClaw может делать сам:

- сгенерировать план изменений;
- сгенерировать код/скрипты;
- предложить команды установки и деплоя;
- в некоторых конфигурациях выполнить разрешенные команды.

Что все равно нужно контролировать извне:

- права доступа к хосту (файлы, сервисы, пакеты, sudo);
- подтверждение рискованных операций;
- проверка результата (логи, health, smoke-test).

Практически это работает как полуавтоматический pipeline:

1. PicoClaw готовит изменения.
2. Пользователь/политика подтверждает.
3. Изменения применяются ограниченным рантаймом.
4. Выполняется проверка и фиксируется журнал действий.

## Можно ли дать спец-пользователям управление через Telegram/чат

Да, можно. Но безопасно это делается через модель ролей и ограничений, а не через полный shell/root в чате.

Рекомендуемая модель (RBAC + sandbox):

1. Доступ только для явного списка пользователей (`ALLOWED_USERS`).
2. Разделение ролей:
   - `viewer`: только чтение статуса/логов;
   - `operator`: безопасные операции (рестарт сервисов, запуск заранее разрешенных deploy-задач);
   - `admin`: расширение функциональности по регламенту с подтверждением.
3. Никакого произвольного выполнения команд: только allowlist-команды/скрипты.
4. Двухшаговое подтверждение опасных действий: preview -> confirm.
5. Запуск от отдельного системного пользователя, не от `root`.
6. `sudoers` только на узкий набор команд (например, конкретные `systemctl` и deploy-скрипты).
7. Полный audit-log: кто, когда, что запускал, с каким результатом.
8. Для саморасширения - обязательный конвейер:
   - генерация кода;
   - проверка (lint/test/smoke);
   - просмотр diff;
   - применение;
   - перезапуск/проверка.

Антипаттерн:

- давать полные права на хосте через чат без ограничений и журналирования.

Практический вывод для вашего случая: расширять PicoClaw корректно через **слоистую архитектуру** — ядро picoclaw для LLM/каналов и внешние сервисы (Python/shell) для вашей логики, оркестрируя все через systemd + env + конфиги.
