## Problem

Currently, all new Telegram users who contact the bot are placed in `pending` state. They see only a "waiting for approval" message with zero functionality. This creates a poor first impression and blocks demo/onboarding scenarios.

**Goal:** Allow a "limited guest" mode where new (or explicitly granted) users can interact with the bot meaningfully, before full approval.

---

## Concept

### Guest User Definition

A **Limited Guest** is a user who:
- Has sent `/start` (created a registration record), OR been explicitly granted guest status by admin
- Has NOT been approved as a full user
- Gets a **defined subset of features** with **rate limits**
- Can be promoted to Full User by admin at any time
- Sees a different (simpler) main menu

### Guest vs. Other Roles

| | Pending (current) | Limited Guest (planned) | Full User |
|---|---|---|---|
| Chat (LLM) | ❌ | ✅ rate-limited | ✅ |
| Digest | ❌ | ✅ | ✅ |
| Notes/Calendar/Contacts/Docs | ❌ | ❌ | ✅ |
| Voice | ❌ | ❌ | ✅ |
| Profile view | ❌ | ✅ | ✅ |
| Help | ✅ | ✅ | ✅ |
| Request access | ✅ | ✅ | — |

## Registration & Activation Flow

### Automatic Guest (option A — recommended)

When `AUTO_GUEST_ENABLED=1` in `bot.env`:

```
User sends /start
→ Bot asks for name / language
→ Creates registrations.json record with status = "guest"
→ Sends notification to admin
→ User immediately sees guest menu (Chat, Digest, Help, Profile, Request Full Access)
```

Admin panel shows guest count + can promote or block.

### Manual Guest Grant (option B)

Default behavior: new users still go to `pending`.  
Admin can change a pending user to `guest` via Admin Panel → User Management.

---

## Rate Limiting Design

**Storage:** In-memory dict `_guest_message_counts: dict[int, dict]` — counts per `chat_id` per hour/day.  
**Persistence:** reset daily via `_reset_guest_counts_daily()` called from bot startup / cron-style check.

| Limit | Default | Config constant |
|---|---|---|
| Max messages / day | 20 | `GUEST_MSG_DAILY_LIMIT` |
| Max messages / hour | 5 | `GUEST_MSG_HOURLY_LIMIT` |
| Max tokens / response | 500 | `GUEST_MAX_TOKENS` |

When a guest hits the limit:
- Sends a friendly message: "Вы достигли дневного лимита сообщений. Запросите полный доступ → [Запросить доступ]"
- Admin sees the limit-hit event in admin panel (optional)

## Configuration (new constants)

Add to `src/core/bot_config.py`:

```python
AUTO_GUEST_ENABLED     = int(os.environ.get("AUTO_GUEST_ENABLED",     "0"))
GUEST_MSG_DAILY_LIMIT  = int(os.environ.get("GUEST_MSG_DAILY_LIMIT",  "20"))
GUEST_MSG_HOURLY_LIMIT = int(os.environ.get("GUEST_MSG_HOURLY_LIMIT", "5"))
GUEST_MAX_TOKENS       = int(os.environ.get("GUEST_MAX_TOKENS",       "500"))
```

Add to `.env.example`:
```
AUTO_GUEST_ENABLED=0         # Set to 1 to auto-grant guest access to all new registrants
GUEST_MSG_DAILY_LIMIT=20     # Max chat messages per day for guest users
GUEST_MSG_HOURLY_LIMIT=5     # Max chat messages per hour for guest users
GUEST_MAX_TOKENS=500         # Max LLM response tokens for guest users
```

## Out of Scope (won't implement)

- Guest access to voice commands (hardware resource)
- Guest web UI (separate auth flow needed first — see TODO §1 OAuth)
- Guest-specific LLM model (all roles use same configured model)
- Time-limited guest access (auto-expire) — too complex for Phase 1
