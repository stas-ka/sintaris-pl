# 4.3 Remote MCP RAG — Concept & Architecture (For Review)

**Version:** 0.3 (N8N MCP Server replaces custom FastAPI MCP server — 2026-04-23)
**Date:** 2026-04-23
**Status:** 🟡 APPROVED — architecture revised: N8N MCP Server Trigger replaces custom FastAPI service
**Scope:** OpenClaw variant only (TariStation2, TariStation1/SintAItion, VPS-Supertaris). PicoClaw unaffected.
**Parent concept:** [doc/archive/concept/rag-memory-architecture.md §4.3, §6.1, §7 Phase D](../archive/concept/rag-memory-architecture.md)

---

## 1. Goals

1. Provide a **Hybrid Enhanced RAG + Multi-Level Memory** service that runs **outside** the Taris process — on the VPS (`VPS-Supertaris` / `agents.sintaris.net`) and/or in N8N.
2. Expose the service to Taris via the **Model Context Protocol (MCP)**.
3. Bind the remote MCP into Taris **only as a dedicated Skill/Agent** (new `remote_kb` agent). **Existing Taris RAG, document upload, memory, notes, calendar, contacts remain untouched.**
4. Implement **two interchangeable variants** of the remote RAG backend:
   - **Variant 1 — N8N-native**: ingest + retrieve via N8N workflows (Docling node / HTTP nodes / Postgres node / embeddings). Pure workflow-driven.
   - **Variant 2 — Google-grounded**: retrieval is delegated to **Google Programmable Search / Vertex AI Grounding** with local document corpus as a secondary signal.
5. Allow **document import through the Taris UI** (Telegram + Web), but the **actual ingestion, chunking, embedding, storage happens on the remote service**. Taris only uploads the file and stores a reference.
6. Use a **separate PostgreSQL database** for memory, RAG chunks, and conversations (isolated from existing `taris_vps` DB). After benchmarking we may merge back.
7. Benchmark quality and latency using the **Worksafety-superassistant** corpus (≈100 k chunks, ≈200 PDFs; external reference system — see [research §7](../research/rag-memory-extended-research.md#7-reference-architecture--worksafety-superassistant)).

---

## 2. Non-Goals

- No change to `bot_documents.py`, `bot_rag.py`, `bot_memory.py` (if/when built), or any existing `store.*` method. The new path is **strictly additive**.
- No change to PicoClaw (Pi 3/4) — this is OpenClaw/x86_64/VPS only.
- No merger of the existing `documents` / `chunks_fts` / `vec_embeddings` tables with the new RAG DB at this stage.
- No change to existing LLM providers, menus, or agent routing for default chat. Users reach the remote KB only by explicitly invoking the new Skill/Agent.

**Key change from v0.2:** No custom FastAPI service / Docker container on VPS.

File upload (ingest) uses N8N's existing Webhook node (plain HTTP POST, not MCP).

### 4.2 Skill + Agent Binding on OpenClaw (inside Taris)

| Element | Location | Behaviour |
|---|---|---|
| **New agent** `remote_kb` | `src/telegram/bot_agents.py` (or equivalent) | Explicit selection only: command `/kb`, button "🌐 Knowledge Base" in menu, Telegram slash, or Web UI toggle. Never triggered by default router. |
| **New skill** `remote_kb.search(query)` | registered in skill registry (OpenClaw skills API) | Wraps `bot_mcp_client.remote_search()`. Returns ranked chunks + source URIs. |
| **New skill** `remote_kb.ingest(file_id, filename)` | same | Uploads file stream from Telegram/Web to remote `/mcp/ingest`. |
| **New skill** `remote_kb.memory(op, key, value?)` | same | Long-term memory read/write for this agent only. |
| **Existing chat/RAG skills** | untouched | Default agent keeps using local `chunks_fts` / `vec_embeddings`. |
| **Config gate** | `REMOTE_KB_ENABLED` in `bot.env` (default `0`) | When `0`, agent is hidden from menus. |
| **Per-user gate** | `users.remote_kb_allowed` column (role-based) | Only `admin` + roles with explicit grant. |

**Rule:** if `REMOTE_KB_ENABLED=0` **or** `MCP_REMOTE_URL=""`, the Skill and Agent are not registered → zero effect on existing behaviour. This is the safety switch.

### 4.3 Remote Service — Two Variants

Both variants expose the same MCP contract (§5). The VPS operator chooses which is active via env `REMOTE_KB_MODE=n8n|google`.

#### Variant 1 — N8N MCP Server (native)

> N8N workflows are published directly as an MCP server using the **MCP Server Trigger** node.
> No intermediate FastAPI service. No extra container. No extra port.

**MCP Server workflow** (tools exposed to Taris via SSE):

| Tool (MCP name) | N8N flow path | Purpose |
|---|---|---|
| `kb_search` | MCP Server Trigger → embed → Postgres CTE → Code (rerank) → respond | Hybrid vector+BM25 search |
| `kb_memory_get` | MCP Server Trigger → Postgres SELECT → respond | Read memory tier for chat_id |
| `kb_memory_append` | MCP Server Trigger → Postgres INSERT → compaction check → respond | Append + compact |
| `kb_memory_clear` | MCP Server Trigger → Postgres DELETE → respond | Clear memory tier |
| `kb_list_documents` | MCP Server Trigger → Postgres SELECT → respond | List user's docs |
| `kb_delete_document` | MCP Server Trigger → Postgres DELETE CASCADE → respond | Remove doc + chunks |

**Ingest workflow** (separate, plain HTTP POST — binary upload via multipart):

| Stage | Technology | N8N Node |
|---|---|---|
| Entry | `POST /webhook/kb-ingest` (N8N Webhook Trigger) | `Webhook` |
| Parse | **Docling** via `Execute Command` (`python -m docling …`) | `Execute Command` |
| Chunk | recursive 512-char, 50 overlap, section-aware | `Code` |
| Embed | `multilingual-e5-small` (384-dim) via local Ollama `/api/embeddings` | `HTTP Request` → Ollama |
| Store | INSERT into `kb_documents` + `kb_chunks` (pgvector) | `Postgres` |
| Response | `{doc_id, n_chunks, n_embedded, parse_time_ms}` | `Respond to Webhook` |

**Why separate ingest?** MCP tool calls have a small request size limit (JSON payload). Binary file uploads must go via a plain HTTP POST webhook — not via MCP `tools/call`.

**Pros:** Workflows visible in N8N UI, editable without code deploy, no extra container, native MCP protocol.
**Cons:** N8N as critical path; SSE session overhead for each MCP call (mitigated by client-side connection reuse).

#### Variant 2 — Google-grounded

| Stage | Technology |
|---|---|
| Ingest | same ingest path as Variant 1 (keeps local corpus for citation & privacy), but also indexes into Vertex AI Search data store (or skipped if privacy-sensitive) |
| Retrieve | **Google Programmable Search / Vertex AI Grounding** for public-web recall; local pgvector for uploaded corpus |
| Fusion | RRF between Google results (top-k URL+snippet) and local pgvector results |
| Grounding proof | Vertex AI returns citation objects (URI + span) → mapped into MCP response `sources` |
| Response | JSON `{chunks:[…], grounding:[…], latency_ms, mode:"google"}` |

**Pros:** high-quality open-domain recall, official grounding, no embedding maintenance for public web.
**Cons:** cost per query, external dependency, data egress policy, privacy for user uploads must stay in local corpus (Google is public-web only).

**Both variants share:** same N8N MCP Server entry point, same DB schema, same MCP tool names/signatures. Backend is swappable via `REMOTE_KB_MODE` env var. A/B benchmarking runs both side-by-side via benchmark harness (§7).

### 4.4 Data Isolation — Separate Postgres Database

New database `taris_kb` on the same Postgres cluster as `taris_vps`, but **separate role and connection string**.

| Aspect | Existing `taris_vps` | New `taris_kb` |
|---|---|---|
| Purpose | bot data (users, docs, chat_history, contacts, …) | RAG chunks, remote memory, remote conversations |
| Owner role | `taris` | `taris_kb_writer` (new, minimal grants) |
| Consumed by | Taris process | **only** the remote MCP service |
| Taris access | ❌ never direct (MCP only) | — |
| Merge decision | deferred until benchmark complete |

#### Schema (draft, review needed)

```sql
-- Documents registered via /mcp/ingest
CREATE TABLE kb_documents (
    doc_id       UUID PRIMARY KEY,
    owner_chat_id BIGINT NOT NULL,
    title        TEXT NOT NULL,
    mime         TEXT,
    sha256       TEXT UNIQUE,
    source       TEXT,              -- 'taris_upload' | 'web_crawl' | 'manual'
    structure    JSONB,             -- docling outline: chapters/sections
    created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- Structure-aware chunks
CREATE TABLE kb_chunks (
    chunk_id     BIGSERIAL PRIMARY KEY,
    doc_id       UUID REFERENCES kb_documents ON DELETE CASCADE,
    chunk_idx    INT,
    section      TEXT,              -- e.g. "Chapter 3.2 — …"
    text         TEXT NOT NULL,
    tokens       INT,
    embedding    vector(384),       -- pgvector, HNSW index
    fts          tsvector GENERATED ALWAYS AS (to_tsvector('simple', text)) STORED,
    metadata     JSONB,
    created_at   TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX ON kb_chunks USING hnsw (embedding vector_cosine_ops);
CREATE INDEX ON kb_chunks USING gin (fts);
CREATE INDEX ON kb_chunks (doc_id, chunk_idx);

-- Memory (short / middle / long) per (chat_id, agent='remote_kb')
CREATE TABLE kb_memory (
    chat_id      BIGINT,
    tier         TEXT CHECK (tier IN ('short','middle','long')),
    seq          BIGSERIAL,
    role         TEXT,              -- user | assistant | system | summary
    content      TEXT NOT NULL,
    source_ids   BIGINT[],          -- references to earlier rows when tier='middle'|'long'
    tokens       INT,
    created_at   TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (chat_id, tier, seq)
);

-- Conversation log for the remote_kb agent only
CREATE TABLE kb_conversations (
    conv_id      UUID PRIMARY KEY,
    chat_id      BIGINT NOT NULL,
    started_at   TIMESTAMPTZ DEFAULT NOW(),
    last_at      TIMESTAMPTZ DEFAULT NOW(),
    meta         JSONB
);

CREATE TABLE kb_messages (
    msg_id       BIGSERIAL PRIMARY KEY,
    conv_id      UUID REFERENCES kb_conversations ON DELETE CASCADE,
    role         TEXT,
    content      TEXT,
    chunks_used  BIGINT[],          -- → kb_chunks.chunk_id
    grounding    JSONB,             -- Variant 2: Google citations
    latency_ms   INT,
    mode         TEXT,              -- 'n8n' | 'google'
    created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- Query cache (Worksafety pattern)
CREATE TABLE kb_query_cache (
    query_sha256 TEXT PRIMARY KEY,
    mode         TEXT,
    response     JSONB,
    created_at   TIMESTAMPTZ DEFAULT NOW(),
    ttl_sec      INT DEFAULT 3600
);
```

## 5. MCP Contract (MCP protocol over SSE, N8N auth)

> **Transport:** SSE (Server-Sent Events) via N8N's MCP Server Trigger endpoint.
> **Endpoint:** `https://agents.sintaris.net/n8n/mcp/<workflow-path>/sse`
> **Auth:** N8N API key in `X-N8N-API-Key` header (set in `N8N_KB_API_KEY` env var).
> **Client library:** Python `mcp` package (`from mcp import ClientSession; from mcp.client.sse import sse_client`).

### MCP Tools (exposed via N8N MCP Server Trigger workflow)

| Tool name | Input schema | Output schema | Phase |
|---|---|---|---|
| `kb_search` | `{query: str, top_k?: int, mode?: "n8n"\|"google", chat_id?: int}` | `{chunks:[{doc_id,section,text,score,source_uri}], latency_ms, mode}` | 2 |
| `kb_list_documents` | `{chat_id: int}` | `[{doc_id,title,n_chunks,created_at}]` | 2 |
| `kb_delete_document` | `{doc_id: str}` | `{ok: bool}` | 2 |
| `kb_memory_get` | `{chat_id: int, tier: "short"\|"middle"\|"long"}` | `[{role,content,created_at}]` | 2 |
| `kb_memory_append` | `{chat_id: int, role: str, content: str}` | `{ok: bool, compacted: bool}` | 2 |
| `kb_memory_clear` | `{chat_id: int, tier?: str}` | `{ok: bool, deleted: int}` | 2 |

### Ingest endpoint (plain HTTP POST, separate from MCP)

File upload is **not** an MCP tool — binary data goes via N8N's Webhook node directly:

```
POST https://agents.sintaris.net/n8n/webhook/kb-ingest
Content-Type: multipart/form-data
Authorization: Bearer <N8N_KB_TOKEN>

Fields: chat_id, file (binary), filename, mime
Response: {doc_id, n_chunks, n_embedded, parse_time_ms}
```

Errors: HTTP 4xx/5xx from N8N webhook; circuit breaker in `bot_mcp_client.py` handles 3-strike open for both MCP and HTTP paths.

## 6. Document Import Flow (UI in Taris, Work on Remote)

```
 User in Telegram / Web UI
      │  "📂 Upload to remote KB"
      ▼
 bot_agents.remote_kb.ingest()
      │  stream file bytes (chunked HTTP)
      ▼
 Remote /mcp/ingest (FastAPI)
      │  save bytes to /srv/kb/uploads/<sha>.bin
      ▼
 Variant 1:  N8N webhook "kb-ingest"
             Docling → chunks → embed → COPY into kb_chunks
 Variant 2:  background worker (APScheduler) same pipeline + optional Vertex AI index push
      │
      ▼
 Response to Taris: {doc_id, n_chunks, n_embedded, parse_time_ms}
      │
      ▼
 Taris shows receipt in chat (no local document row)
```

**Key constraint:** Taris does **not** touch its own `documents` table for remote-KB uploads. The uploaded file never enters local FTS5. This keeps the two systems independent.

## 7. Benchmarking — Worksafety Corpus

Goal: prove that the remote service matches or exceeds local Taris RAG quality on a domain corpus with well-known ground truth.

| Item | Decision |

## 8. Security & Ops

| Concern | Decision |
|---|---|
| AuthN | Bearer token shared between Taris and remote. Rotated via `.env`. |
| AuthZ | Per-user role flag `remote_kb_allowed`. Admin-only at first. |
| Data scope | Each chat_id only sees its own `kb_documents` (WHERE `owner_chat_id`); admin can list all. |
| PII in logs | `kb_messages.content` retained 30 days then truncated unless user opts in. Telemetry is **remote-only** — no write-back to local `rag_log`. |
| Rate limits | N8N built-in rate limiting + nginx upstream limit. |
| TLS | nginx on `agents.sintaris.net`; N8N's existing nginx config already handles `/n8n/`. |
| Secrets | Never commit. Placeholders in `.env.example`: `MCP_REMOTE_URL` (N8N MCP SSE URL), `N8N_KB_API_KEY` (N8N API key), `N8N_KB_TOKEN` (webhook Bearer), `KB_PG_DSN`, `GOOGLE_CSE_KEY`, `GOOGLE_CSE_CX`, `VERTEX_PROJECT`, `VERTEX_LOCATION`. |
| Variant 2 / uploads | User-uploaded documents NEVER pushed to Vertex AI or Google. Google-grounded mode is public-web only; private corpus stays exclusively in `kb_chunks` (pgvector). |
| VPS-Supertaris rule | All deployments require **explicit per-step confirmation** from user (per AGENTS.md). This includes: nginx edit, Postgres DDL, N8N workflow import, service installs. |

## 10. Resolved Design Decisions (2026-04-23)

| # | Question | Decision |
|---|---|---|
| 1 | Variant build order | **N8N first**, Google second. Sequential, not parallel. |
| 2 | Memory scope | **Separate** (`taris_kb` only) for now. Merge into main Taris memory is **opt-in** after Phase 6 benchmark. |
| 3 | Corpus privacy (Variant 2) | User uploads **never** sent to Vertex AI / Google. Google grounding is **public-web only**. Uploaded corpus stays exclusively in `kb_chunks` (pgvector). |
| 4 | `chat` tool | **Return chunks only.** Taris calls its own LLM. Full-turn mode (`REMOTE_KB_FULL_TURN=1`) added later as opt-in option. |
| 5 | Docling placement | **Inside N8N via `Execute Command` node** (`python -m docling …`). No separate sidecar. |
| 6 | Embedding model | **`multilingual-e5-small` (384-dim) via local Ollama `/api/embeddings`** — consistent with existing OpenClaw stack, supports RU/DE/EN. |
| 7 | DB merge | **No merge** — `taris_kb` stays isolated from `taris_vps`. Operational clarity > consolidation. |
| 8 | Worksafety corpus | Pull from **`stas-ka/Worksafety-superassistant` on GitHub**. PDFs at `/srv/kb/benchmark/worksafety/` on VPS. **Copy existing Worksafety N8N workflows to VPS N8N** as reference baseline for benchmark. |
| 9 | Skill / menu label | **"🌐 Remote Knowledge Base"** (`remote_kb`). i18n keys: `ru="🌐 Удалённая база знаний"`, `en="🌐 Remote Knowledge Base"`, `de="🌐 Remote-Wissensbasis"`. |
| 10 | Telemetry | **Remote-only**: written into `kb_messages` on the remote service only. Local `rag_log` is **not** updated from this path. |
| 11 | MCP server implementation | **N8N MCP Server Trigger** replaces custom FastAPI service. N8N workflows ARE the MCP server. No Docker container. No extra port. Taris uses Python `mcp` library (SSE client). |
| 12 | File ingest transport | **N8N Webhook** (plain HTTP POST multipart). MCP `tools/call` is not used for binary uploads — JSON payload size limits make it impractical. |

## 11. Acceptance Criteria (for the eventual Phase 7 rollout)

> ℹ️ AutoResearch evaluation (§12) gates the go/no-go decision between Phase 6 and Phase 7.

- [ ] `REMOTE_KB_ENABLED=0` → zero observable change in Taris (all existing tests pass).

### 13.3 Async wrapper pattern (for sync Taris callers)

```python
# In bot_mcp_client.py — wraps async MCP call for sync callers
import asyncio

def query_remote(query: str, chat_id: int, top_k: int | None = None) -> list[dict]:
    try:
        return asyncio.run(_query_remote_async(query, chat_id, top_k))
    except RuntimeError:
        # Already in event loop (e.g. python-telegram-bot async handler)
        import concurrent.futures
        with concurrent.futures.ThreadPoolExecutor(max_workers=1) as ex:
            future = ex.submit(asyncio.run, _query_remote_async(query, chat_id, top_k))
            return future.result(timeout=MCP_TIMEOUT)

async def _query_remote_async(query, chat_id, top_k):
    from mcp.client.sse import sse_client
    from mcp import ClientSession
    url = MCP_REMOTE_URL  # https://agents.sintaris.net/n8n/mcp/<path>/sse
    headers = {"X-N8N-API-Key": N8N_KB_API_KEY}
    async with sse_client(url, headers=headers) as (r, w):
        async with ClientSession(r, w) as session:
            await session.initialize()
            result = await session.call_tool("kb_search", {
                "query": query, "chat_id": chat_id, "top_k": top_k or MCP_REMOTE_TOP_K
            })
            return result.content[0].model_dump()["chunks"]
```

### 12.1 Purpose

Use the **Karpathy AutoResearch** paradigm (3-file loop: `program.md` / `evaluate.py` / `prepare.py`) to autonomously optimize the remote RAG pipeline configuration overnight — without manual grid search. The agent iteratively modifies `evaluate.py`, runs a timed experiment, measures `rag_score`, and keeps only improvements.

### 12.3 Composite Metric: `rag_score`

Adapted from Worksafety multi-signal scoring (§7.2) and AutoResearch's `val_bpb` principle:

```python
rag_score = (
    0.35 * recall_at_5        # fraction of ground-truth answers in top-5 chunks
  + 0.25 * mrr_at_10          # mean reciprocal rank
  + 0.15 * faithfulness        # LLM-judge: does answer match source? (0..1)
  + 0.15 * (1 / latency_norm)  # normalised: 1.0 at 500 ms, 0.0 at 5000 ms
  + 0.10 * (1 / cost_norm)     # 0 for local Ollama; scaled for cloud API cost
)
```

`latency_norm = clamp(latency_p95_ms / 5000, 0.001, 1.0)`
`cost_norm = clamp(cost_per_query_eur * 100, 0.001, 1.0)` (€0 for fully local)

### 12.6 `evaluate.py` — Benchmark Runner Skeleton

```python
"""
evaluate.py — AutoResearch RAG benchmark runner.
The AI agent modifies only the CONFIG dict below (TUNABLE section).
"""
import json, time, hashlib, statistics
from pathlib import Path
import httpx

# ─── TUNABLE — agent modifies this block only ────────────────────────────────
CONFIG = {
    "chunk_size":         512,
    "chunk_overlap":      50,
    "retrieval_mode":     "hybrid_rrf",   # vector_only | bm25_only | hybrid_rrf
    "top_k":              5,
    "rrf_k":              60,
    "signal_weights": {"vector": 0.35, "bm25": 0.30, "section": 0.15,
                       "recency": 0.10, "owner": 0.10},
    "cache_ttl_sec":      1800,
    "memory_tier":        "short_middle",  # short_only | short_middle | full_3tier
    "memory_window":      15,
}
# ─────────────────────────────────────────────────────────────────────────────

MCP_URL   = "https://agents.sintaris.net/mcp/search"   # set in env
TOKEN     = open(Path.home()/".taris/kb_token").read().strip()
QA_PAIRS  = json.loads((Path(__file__).parent/"qa_pairs.json").read_text())

def search(query: str, top_k: int) -> list[dict]:
    resp = httpx.post(MCP_URL, json={"query": query, "top_k": top_k,
                                      "config": CONFIG},
                      headers={"Authorization": f"Bearer {TOKEN}"}, timeout=10)
    return resp.json().get("chunks", [])

def evaluate_faithfulness(answer: str, chunk_texts: list[str]) -> float:
    """LLM-judge: does the answer faithfully match the retrieved chunks? 0..1"""
    # Call local Ollama qwen3.5 with judge prompt — cost = 0
    import ollama
    prompt = (f"Given the retrieved passages below, rate 0.0–1.0 how faithfully "
              f"the answer is grounded in them.\n\nPassages:\n" +
              "\n---\n".join(chunk_texts[:3]) +
              f"\n\nAnswer: {answer}\n\nScore (0.0–1.0):")
    r = ollama.generate(model="qwen3.5:latest", prompt=prompt)
    try:    return float(r["response"].strip().split()[0])
    except: return 0.5

latencies, recall5s, reciprocal_ranks, faithfulness_scores = [], [], [], []

for qa in QA_PAIRS:
    t0 = time.perf_counter()
    chunks = search(qa["question"], CONFIG["top_k"])
    latencies.append((time.perf_counter() - t0) * 1000)

    texts = [c["text"] for c in chunks]
    # Recall@5: does any chunk contain the expected answer?
    recall5s.append(1.0 if any(qa["answer_hint"].lower() in t.lower() for t in texts) else 0.0)
    # MRR@10
    rr = 0.0
    for i, t in enumerate(texts[:10]):
        if qa["answer_hint"].lower() in t.lower(): rr = 1/(i+1); break
    reciprocal_ranks.append(rr)
    faithfulness_scores.append(evaluate_faithfulness(qa["answer_hint"], texts))

latency_p95 = sorted(latencies)[int(len(latencies)*0.95)]
latency_norm = min(max(latency_p95 / 5000, 0.001), 1.0)

rag_score = (0.35 * statistics.mean(recall5s)
           + 0.25 * statistics.mean(reciprocal_ranks)
           + 0.15 * statistics.mean(faithfulness_scores)
           + 0.15 * (1 / latency_norm)
           + 0.10 * 1.0)  # cost_norm = 1.0 (fully local Ollama)

result = {
    "rag_score":     round(rag_score, 4),
    "recall_at_5":   round(statistics.mean(recall5s), 4),
    "mrr_at_10":     round(statistics.mean(reciprocal_ranks), 4),
    "faithfulness":  round(statistics.mean(faithfulness_scores), 4),
    "latency_p50":   round(statistics.median(latencies), 1),
    "latency_p95":   round(latency_p95, 1),
    "config":        CONFIG,
}
print(json.dumps(result, indent=2))
```
