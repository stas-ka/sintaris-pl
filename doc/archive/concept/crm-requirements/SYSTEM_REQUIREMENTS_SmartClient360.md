## 1. ОБЩИЕ ПОЛОЖЕНИЯ

### 1.1 Назначение системы

SmartClient 360 — **многопользовательская (multi-tenant) SaaS-платформа** для автоматизации клиентского цикла медицинских центров, клиник и сервисных организаций здоровья. Платформа предоставляется в виде услуги (SaaS) разработчиком/провайдером (далее — **Оператор платформы**) клиентским организациям (далее — **Организации-клиенты**), которые используют систему для работы со своими пациентами/клиентами (далее — **Конечные пользователи**).

### 1.2 Участники взаимодействия

| Роль | Описание |
|------|----------|
| **Оператор платформы** | Разработчик и владелец платформы SmartClient 360. Не является оператором персональных данных конечных пользователей. |
| **Организация-клиент** | Юридическое лицо (медцентр, клиника и т.д.), подписавшееся на платформу. Является **оператором персональных данных** (в смысле 152-ФЗ) своих пациентов. Имеет изолированное хранилище данных. |
| **Администратор организации** | Сотрудник организации-клиента. Управляет настройками, записями, контентом. |
| **Врач / Специалист** | Сотрудник организации-клиента. Работает с карточками пациентов, ведёт переписку, отправляет рекомендации. |
| **Руководитель** | Владелец / руководитель организации-клиента. Видит аналитику, KPI, дашборды. |
| **Конечный пользователь (Пациент)** | Клиент организации. Взаимодействует с системой через веб-портал (и опционально через мессенджер). |

### 1.3 Исходные данные и ограничения

- Пилотная площадка: нутрициологический центр (~500+ клиентов).
- Целевые сегменты: нутрициология, косметология, стоматология, психология, фитнес, ветеринария, салоны красоты.
- Использование Telegram **не является обязательным** и не должно быть единственным каналом взаимодействия с конечным пользователем. **Основной канал — веб-интерфейс**.
- Конечные пользователи (пациенты) должны получать доступ через веб-браузер без необходимости установки приложений.
- Все персональные данные должны храниться **исключительно в  организации-клиента** и не должны копироваться или храниться на инфраструктуре Оператора платформы.

## 3. СООТВЕТСТВИЕ 152-ФЗ

### 3.1 Требования законодательства

| Требование 152-ФЗ | Как обеспечивается в архитектуре |
|-------------------|---------------------------------|
| Хранение ПДн граждан РФ **на серверах в РФ** | (серверы в РФ). S3: VK Cloud / Yandex Object Storage (серверы в РФ). |
| **Оператор ПДн** — юридическое лицо | Организация-клиент подписывает соглашение об обработке ПДн. Оператор платформы выступает **обработчиком**, а не оператором. |
| **Согласие на обработку** ПДн | При регистрации в веб-портале конечный пользователь даёт явное согласие. Факт согласия хранится в CRM организации-клиента. |
| **Право на удаление** (право быть забытым) | Workflow удаления: по запросу — очистка данных . Оператор платформы не хранит ПДн и не участвует в удалении. |
| **Уведомление Роскомнадзора** | Ответственность организации-клиента как оператора ПДн. Оператор платформы предоставляет шаблон уведомления. |
| **Договор на обработку** ПДн | Оператор платформы заключает договор на обработку ПДн с каждой организацией-клиентом. |

### 3.2 Архитектурные гарантии 152-ФЗ

- **REQ-152-01:** Персональные данные конечных пользователей (ФИО, дата рождения, телефон, диагностические данные) хранятся **исключительно в CRM организации-клиента** — в Битрикс24 Cloud (серверы РФ) или в согласованном On-Premise решении.
- **REQ-152-02:** Инфраструктура Оператора платформы обрабатывает данные в **stateless-режиме** — не кэширует и не сохраняет ПДн в собственных базах данных.
- **REQ-152-03:** При передаче данных в AI-модuls (LLM) — используется **обезличивание**: ФИО, телефон, email заменяются на внутренние ID клиента до отправки в GigaChat/YandexGPT API. Оба провайдера располагают серверами в РФ и соответствуют 152-ФЗ.
- **REQ-152-04:** Файлы (анализы, PDF-документы) хранятся в облачном хранилище на серверах в РФ , привязанном к организации-клиенту.
- **REQ-152-05:** Логи операций Оператора платформы содержат **только технические идентификаторы** (record_id, tenant_id) без ПДн.

### 3.3 Распределение ролей по 152-ФЗ

```

│ ОПЕРАТОР ПЛАТФОРМЫ  │         │    ОРГАНИЗАЦИЯ-КЛИЕНТ          │
│ (SmartClient 360)   │         │    (Медицинский центр)          │
│                     │         │                                 │
│ Роль: ОБРАБОТЧИК    │◄──────► │ Роль: ОПЕРАТОР ПДн             │
│ (ст.6 п.3 152-ФЗ)  │ Договор │ (ст.6 152-ФЗ)                  │
```

## 7. ВЕБ-ДОСТУП И КЛИЕНТСКИЙ ПОРТАЛ

### 7.1 Обоснование перехода на веб-первичность

В связи с возможными ограничениями использования Telegram в РФ основным каналом взаимодействия должен быть **веб-интерфейс**. Telegram и другие мессенджеры — дополнительные, опциональные каналы.

### 7.2 Требования к веб-порталу для конечных пользователей (пациентов)

- **REQ-WEB-01:** Веб-портал должен быть доступен по HTTPS без установки дополнительного программного обеспечения.
- **REQ-WEB-02:** Веб-портал должен быть адаптивным (responsive) для мобильных устройств и планшетов.
- **REQ-WEB-03:** Веб-портал должен поддерживать PWA (Progressive Web App) для возможности добавления на экран смартфона.
- **REQ-WEB-04:** Каждая организация-клиент имеет **изолированный веб-портал** (отдельный субдомен или конфигурация).
- **REQ-WEB-05:** Веб-портал должен поддерживать **брендирование организации** (лого, цвета, название).
- **REQ-WEB-06:** Стек веб-портала: **Next.js + TypeScript** (SSR для SEO и скорости) или React SPA.

### 7.3 Функциональность веб-портала для пациентов

| # | Функция | Описание | Этап |
|---|---------|----------|------|
| P1 | Регистрация/Авторизация | Email + OTP или номер телефона + SMS OTP | 0–1 |
| P2 | Онлайн-запись | Выбор врача, даты, времени из расписания | 1 |
| P3 | Мои записи | Просмотр текущих и прошлых визитов | 1 |
| P4 | Уведомления | Push-уведомления браузера + email о записях и напоминаниях | 1 |
| P5 | Заполнение анкеты | Адаптивная анкета перед визитом | 1 |
| P6 | Чат с врачом | Текстовый чат (интегрированный с CRM), история переписки | 2 |
| P7 | Мои рекомендации | Просмотр и скачивание PDF-плана лечения | 2 |
| P8 | Оценка визита (NPS) | Форма обратной связи после визита | 2 |
| P9 | Клуб здоровья | Просмотр контента, статей, акций | 2 |
| P10 | Загрузка анализов | Загрузка фото/PDF анализов | 3 |
| P11 | AI-консультант | Онлайн-чат с AI-помощником в рамках портала | 3 |
| P12 | Личный кабинет | Профиль, история визитов, документы | 1 |

### 7.4 Веб-интерфейс для сотрудников организации

| Роль | Интерфейс | Описание |
|------|-----------|----------|
| Администратор |  Web + Панель управления | через браузер |
| Врач |  Web + Панель врача | Работа с карточками, переписка |
| Руководитель |  + Metabase/DataLens | все + BI-дашборды |

### 7.5 Telegram как дополнительный канал (опционально)

- При наличии доступа Telegram может использоваться как дополнительный канал для уведомлений.
- **REQ-TG-01:** Все функции системы должны быть доступны через веб без Telegram.
- **REQ-TG-02:** Telegram-компонент должен быть изолирован в отдельном наборе workflow так, чтобы его отключение не влияло на остальную систему.
- **REQ-TG-03:** Пользователи, не использующие Telegram, должны получать все уведомления по email.
- Инструмент Salebot может оставаться для сегментов, где Telegram доступен.

## 8. АВТОРИЗАЦИЯ И РОЛИ

### 8.1 Требования к системе аутентификации

- **REQ-AUTH-01:** Применяется единая система аутентификации для всех компонентов (веб-портал, API, административные панели).
- **REQ-AUTH-02:** Аутентификация конечных пользователей (пациентов) — через **email + OTP** или **номер телефона + SMS OTP**. Пароли опциональны.
- **REQ-AUTH-03:** Аутентификация сотрудников организаций — стандартная авторизация  (email + пароль, LDAP/SSO при необходимости).
- **REQ-AUTH-04:** Веб-портал использует JWT-токены. Срок жизни access-token — 1 час. Refresh-token — 30 дней.
- **REQ-AUTH-05:** Поддержка многофакторной аутентификации для ролей «Руководитель» и «Администратор».
- **REQ-AUTH-06:** При входе в систему **определяется тенант** пользователя (по субдомену или по параметру организации в URL) и выдаётся токен с `tenant_id`.

### 8.2 Матрица ролей и прав доступа

| Объект / Функция | Пациент (конечный пользователь) | Администратор | Врач | Руководитель |
|-----------------|--------------------------------|--------------|------|-------------|
| Записи к врачу | Только собственные (CRUD) | Все (CRUD) | Просмотр своих | Просмотр всех |
| Карточка пациента (ФИО, контакты) | Только своя (чтение) | Только запись (чтение) | Полная (чтение + аннотации) | Полная |
| Анкета и медданные | Только своя (заполнение) | Только статус | Полный доступ | Просмотр агрегировано |
| Переписка с врачом | Только свою (чтение + письмо) | **НЕТ ДОСТУПА** | Полная (своя с клиентами) | Чтение (аудит) |
| AI-отчёт врача | НЕТ | НЕТ | Только свои | Просмотр |
| Протокол / рекомендации | Только свои (чтение) | НЕТ | Полный доступ (создание) | Просмотр |
| Клуб здоровья | Просмотр | Публикация | НЕТ | Полный |
| Задачи эскалации | НЕТ | CRUD (свои) | Просмотр | Полный |
| Расписание | Просмотр свободных слотов | CRUD | Просмотр своего | Полный |
| Дашборд KPI | НЕТ | НЕТ | НЕТ | Полный |
| Управление сотрудниками | НЕТ | НЕТ | НЕТ | Полный |
| Настройки системы | НЕТ | Частично | НЕТ | Полный |
| Реферальная программа | Просмотр своей | Просмотр | НЕТ | Полный |
| Загрузка анализов | Только свои | НЕТ | Просмотр | НЕТ |

### 8.3 Изоляция ролей между тенантами

- **REQ-AUTH-07:** Вход пользователя возможен только в контексте **своего тенанта** (организации).
- **REQ-AUTH-08:** Токен содержит `tenant_id`, `user_id`, `role`. Все API-запросы валидируются по `tenant_id`.
- **REQ-AUTH-09:** Переключение между тенантами в рамках одной сессии **недопустимо** без повторной аутентификации.
- **REQ-AUTH-10:** Сотрудник организации A не может получить доступ к данным организации B даже если технически знает URL.

## 11. ОПИСАНИЕ ПОДСИСТЕМ

### 11.1 Веб-портал (Patient Portal)

**Назначение:** Основной интерфейс для конечных пользователей (пациентов). Заменяет Telegram-бот как основной канал.

**Функции:**
- Регистрация и авторизация пациентов (email OTP / телефон OTP).
- Онлайн-запись к врачу с выбором специалиста, даты и времени.
- Просмотр и управление своими записями (перенос, отмена).
- Заполнение анкеты перед визитом.
- Чат с врачом (история переписки).
- Получение и просмотр рекомендаций/протоколов (PDF).
- Обратная связь после визита (NPS).
- Раздел «Клуб здоровья» (контент, акции).
- Загрузка анализов (Этап 3+).
- Чат с AI-консультантом (Этап 3+).
- Каталог товаров и оформление подписки (Этап 4+).
- Push-уведомления (напоминания о визитах, сообщения от врача).
- Брендирование организации (лого, цвета).

### 11.2 Оркестратор автоматизации

**Назначение:** Центральный узел интеграции всех систем. Реализует бизнес-логику без написания кода (визуальные workflow).

**Функции:**
- Приём webhook-событий от Web Portal API, Битрикс24, Salebot.
- Маршрутизация и обработка событий (записи, напоминания, чат, NPS).
- Cron-задачи (напоминания, повторные визиты, рассылки, прогнозы оттока).
- Интеграция с центральным приложением, Calendar, AI Engine, SMS/Email провайдерами.
- Изолированный набор workflows per тенант (credentials изоляция).
- Логирование всех операций (без ПДн).

**Ключевые особенности для multi-tenant:**
- Один экземпляр  обслуживает все тенанты.
- Для каждого тенанта — отдельные credential-объекты и копии шаблонных workflows.
- Tenant ID передаётся в каждом webhook-запросе и используется для маршрутизации в нужные credentials.

### 11.3 Центральное приложение один портал per тенант

**Назначение:** Хранение всех бизнес-данных организации-клиента, включая персональные данные пациентов.

**Функции (задействованные в SmartClient 360):**
- Контакты: карточки пациентов (ПДн, статусы, история).
- Сделки: визиты (записи, анкеты, статусы, ссылки на документы).
- Задачи: эскалации для администратора (неподтверждённые записи, жалобы).
- Календарь: расписание врачей, свободные слоты.
- Открытые каналы: переписка пациент↔врач с историей.
- Пользовательские поля: Telegram ID, lifecycle status, NPS, compliance%, churn_risk, bonus_balance.
- Wwebhooks (исходящие): при изменении статуса, добавлении примечания → в n8n.
- REST API: CRUD операции, аутентификация через OAuth2.
- Управление ролями и правами доступа (встроенный механизм Битрикс24).

**Данные НЕ хранятся у Оператора платформы.** Все ПДн — в облаке клиента на серверах в РФ.

### 11.5 Файловое хранилище

**Назначение:** Хранение файлов, связанных с каждым тенантом (PDF, анализы, аватары).

- Провайдер: **VK Cloud Object Storage** (основной) или **Yandex Object Storage** — оба расположены в РФ, соответствуют 152-ФЗ.
- Изоляция: отдельный bucket для каждого тенанта (`tenant-{id}-files`).
- Права доступа: файлы доступны только через подписанные URL (presigned URL) с ограниченным сроком действия.
- Шифрование: SSE (server-side encryption) обязательно.
- Хранимые объекты: фото/PDF анализов, PDF-рекомендации, AI-отчёты, аватары врачей.

### 11.6 BI-дашборды (Metabase, Этап 4+)

**Назначение:** Аналитика для руководителей организаций-клиентов.

- Open-source (AGPL), self-hosted (Docker).
- Подключается к PostgreSQL с агрегированными (обезличенными) данными из CRM через n8n.
- KPI: записи/день, конверсия, возвратность, NPS, прогноз оттока, LTV.
- Права: каждый портал Metabase — только данные своего тенанта.

## 14. УПРАВЛЕНИЕ ЖИЗНЕННЫМ ЦИКЛОМ ТЕНАНТОВ

### 14.1 Состояния тенанта

```
[Регистрация] → [Онбординг] → [Активный]
                                   │
                         ┌────────┴────────┐
                         │                 │
                   [Приостановлен]    [Деактивирован]
                         │                 │
                   [Активирован]     [Удалён (по запросу)]
```

### 14.2 Процессы управления тенантом

| Событие | Действие системы | Время исполнения |
|---------|-----------------|-----------------|
| Активация нового тенанта | Создание конфига, копирование WF-шаблонов, настройка credentials | 3–4 часа (Этап 5: 30 мин авто) |
| Смена тарифа (апгрейд) | Включение дополнительных компонентов (WF, ChromaDB, Engine) | 1–2 часа |
| Смена тарифа (даунгрейд) | Деактивация выключаемых WF, уведомление тенанта | 1 час |
| Приостановка (неоплата) | Деактивация всех WF, блокировка API-доступа. Данные в Битрикс24 сохраняются | Немедленно |
| Восстановление | Активация WF, восстановление API-доступа | 15 мин |
| Удаление тенанта | Удаление WF и credentials в n8n, удаление ChromaDB-коллекции, удаление S3-bucket. **Данные в Битрикс24 — на усмотрение организации.** | По запросу |

## 15. БЕЗОПАСНОСТЬ И ЗАЩИТА ДАННЫХ

### 15.1 Меры безопасности на уровне транспорта

- **REQ-SEC-01:** Все соединения — только TLS 1.3+. HTTP → автоматический редирект на HTTPS.
- **REQ-SEC-02:** HSTS (HTTP Strict Transport Security) включён для всех доменов.
- **REQ-SEC-03:** Nginx как reverse proxy. Открытые порты: только 80 (редирект) и 443.

### 15.2 Меры безопасности на уровне приложения

- **REQ-SEC-04:** API-запросы к n8n webhook защищены **HMAC-подписью** (секрет per тенант).
- **REQ-SEC-05:** JWT-токены подписаны HMAC-SHA256. Короткий срок жизни access-token (1 час).
- **REQ-SEC-06:** Rate limiting на API Gateway (Nginx / middleware): максимум 100 запросов/мин на IP.
- **REQ-SEC-07:** n8n закрыт от публичного доступа (только localhost или VPN). Управление только через SSH/VPN Оператором.
- **REQ-SEC-08:** Credentials в n8n хранятся в зашифрованном vault (N8N_ENCRYPTION_KEY).

### 15.3 Безопасность данных

- **REQ-SEC-09:** Персональные данные **не хранятся** в SmartClient Engine, n8n, Redis, Web Portal DB.
- **REQ-SEC-10:** При передаче данных анкеты в AI Engine применяется **автоматическое обезличивание** (замена ФИО/телефон на `patient_{{id}}`).
- **REQ-SEC-11:** Файлы в S3 доступны только через **presigned URL** со сроком действия 1 час.
- **REQ-SEC-12:** Ежедневные резервные копии конфигурации n8n и SmartClient Engine (без ПДн) в S3 Оператора.
- **REQ-SEC-13:** Логи содержат только `tenant_id`, `record_id`, технические метрики. Без ФИО, телефонов, email.

### 15.4 Управление инцидентами

- **REQ-SEC-14:** Мониторинг доступности (Uptime Kuma / Grafana) с алертами при недоступности компонентов > 2 минут.
- **REQ-SEC-15:** SLA доступности: 99,5% (не более 3,6 часа downtime в месяц).
- **REQ-SEC-16:** Процедура уведомления тенантов при инцидентах, затрагивающих доступность (Email в течение 15 мин).

## 16. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 16.1 Производительность

| Метрика | Требование |
|---------|-----------|
| Время загрузки веб-портала (LCP) | ≤ 2,5 сек при 3G |
| Время ответа API (P95) | ≤ 300 мс для операций без AI |
| Время ответа AI (P95) | ≤ 15 сек |
| Одновременные пользователи на тенант | ≥ 500 |
| Суммарно одновременных пользователей | ≥ 5 000 (при 10+ тенантах) |
| Webhook обработка очереди | ≤ 5 сек от получения до выполнения действия |

### 16.2 Надёжность

| Метрика | Требование |
|---------|-----------|
| SLA доступности платформы | ≥ 99,5%/мес |
| SLA доступности Web Portal | ≥ 99,5%/мес |
| RTO (Recovery Time Objective) | ≤ 1 час при инциденте |
| RPO (Recovery Point Objective) | ≤ 24 часа (ежедневные бэкапы) |
| Потеря уведомлений | 0% (retry logic в n8n + очередь Redis) |

### 16.3 Масштабируемость

| Параметр | Требование |
|----------|-----------|
| Количество тенантов | ≥ 100 без изменения архитектуры |
| Количество конечных пользователей per тенант | ≥ 10 000 |
| Хранилище per тенант (S3) | ≥ 100 ГБ |
| workflow executions/мес | ≥ 1 000 000 (community edition не ограничен) |
