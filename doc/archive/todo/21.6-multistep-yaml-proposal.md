# Proposal: Multi-Step Flows in YAML Screen DSL

**Version:** Draft · **Date:** 2026-03-22  
→ Parent: [TODO §21](../../TODO.md) · [Screen DSL spec](21-screen-dsl-loader.md) · [Dev patterns §19](../dev-patterns.md)

---

## 1. Problem Statement

The YAML Screen DSL currently handles **static screens** — menus, info cards, profile views, note detail. Ten YAML screens exist and work well.

However, the bot has **six complex multi-step flows** that remain 100% Python:

| Flow | Module | Steps | Pattern |
|---|---|---|---|
| Calendar add event | `bot_calendar.py` | prompt → LLM parse → confirm card (single/multi) → save | LLM-parse-then-confirm |
| Calendar edit field | `bot_calendar.py` | select field → prompt value → LLM re-parse → re-confirm | Edit sub-flow |
| Calendar console | `bot_calendar.py` | free text → LLM intent classify → route to add/query/delete/edit | Intent router |
| Contact add | `bot_contacts.py` | name → phone → email → notes (skip any) → save | Sequential wizard |
| Mail setup | `bot_mail_creds.py` | consent → provider → email → password → IMAP test → save | Sequential wizard |
| Error protocol | `bot_error_protocol.py` | name → collect (text/voice/photo) → send/cancel | Open-ended collector |

These flows share common patterns but cannot be expressed in today's YAML format because they require **runtime state transitions**, **dynamic data population**, and **async operations** (LLM calls, IMAP tests).

---

## 2. Current DSL Capabilities vs. Requirements

| Capability | DSL today | Multi-step requirement |
|---|---|---|
| Static button layout | ✅ | ✅ Sufficient for individual step screens |
| Role-filtered visibility | ✅ | ✅ |
| Variable substitution | ✅ | ✅ Needed: `{event_title}`, `{step_n_of_m}` |
| Conditional visibility | ✅ `visible_if` (variable truthy check) | ✅ Needed: show/hide "Skip" button per step |
| Confirmation gate | ✅ `confirm` widget | ✅ Sufficient for single-event confirm |
| Dynamic lists | ❌ | ⚠️ event list, note list, contact list |
| ForceReply text input | ❌ | ⚠️ Nearly every wizard step |
| State transitions | ❌ | ⚠️ `on_submit → next_screen` |
| LLM async calls | ❌ | ❌ Must stay in Python |
| Media upload handling | ❌ | ❌ Must stay in Python |

---

## 3. Proposed Architecture: Hybrid Screen Pattern

**Principle:** YAML defines the *layout and text* of each step; Python handles the *logic and transitions*. This is NOT a YAML-based state machine — it keeps flow control in Python where it belongs.

### 3.1 Pattern A — YAML Layout Screens for Wizard Steps

Each step of a multi-step flow gets its own YAML screen file. The Python handler loads the appropriate screen, populates variables, and renders it.

```
src/screens/
  cal_confirm.yaml         # Calendar: single-event confirmation card
  cal_confirm_multi.yaml   # Calendar: multi-event "N of M" confirmation
  cal_event_detail.yaml    # Calendar: saved event detail with action buttons
  cal_add_prompt.yaml      # Calendar: "describe your event" prompt
  contact_add_step.yaml    # Contact: per-field prompt screen (dynamic field name)
  mail_consent.yaml        # Mail: GDPR consent gate
  mail_provider.yaml       # Mail: provider selection
  errp_collect.yaml        # Error protocol: collection status + send/cancel
```

**Example: `cal_confirm.yaml`**

```yaml
title: "📅 {event_title}"
parse_mode: Markdown

widgets:
  - type: card
    title: "{event_title}"
    body: "🕒 {event_dt}\n⏰ {remind_text}"

  - type: button_row
    buttons:
      - label_key: cal_btn_save
        action: cal_confirm_save
        style: primary
      - label_key: btn_cancel
        action: menu_calendar
        style: danger

  - type: button_row
    buttons:
      - label_key: cal_btn_edit_title
        action: cal_confirm_edit_title
      - label_key: cal_btn_edit_dt
        action: cal_confirm_edit_dt
      - label_key: cal_btn_edit_remind
        action: cal_confirm_edit_remind

  - type: button_row
    buttons:
      - label_key: cal_btn_tts
        action: cal_confirm_tts
```

**Python handler (unchanged flow control):**

```python
def _show_cal_confirm(chat_id: int) -> None:
    state = _pending_cal[chat_id]
    lang = _st._user_lang.get(chat_id, "ru")

    screen = load_screen("screens/cal_confirm.yaml",
        user=_screen_ctx(chat_id),
        variables={
            "event_title": _escape_md(state["title"]),
            "event_dt":    _format_dt(state["dt_iso"], lang),
            "remind_text": _format_remind(state.get("remind_before_min", 15), lang),
        },
        t_func=_t_by_lang,
    )
    render_screen(screen, chat_id, bot)
```

**Benefits:**
- All user-visible text moves to YAML + `strings.json` → easy i18n
- Layout changes don't require Python code changes
- Flow logic stays in Python → no over-engineered YAML state machine
- Zero new DSL features needed

### 3.2 Pattern B — Dynamic List via Pre-built Markdown

For screens with dynamic data lists (event list, contact list), the Python handler builds a Markdown string and injects it as a variable:

```yaml
# cal_menu.yaml
title_key: cal_menu_title
parse_mode: Markdown

widgets:
  - type: markdown
    text: "{events_list}"
    visible_if: has_events

  - type: markdown
    text_key: cal_no_events
    visible_if: no_events

  - type: button_row
    buttons:
      - label_key: cal_btn_add
        action: cal_add
      - label_key: cal_btn_console
        action: cal_console

  - type: button_row
    buttons:
      - label_key: btn_back
        action: menu
```

```python
def _handle_calendar_menu(chat_id: int) -> None:
    events = _cal_load(chat_id)
    has = bool(events)
    events_md = _format_event_list_md(events, chat_id)  # existing helper

    screen = load_screen("screens/cal_menu.yaml",
        user=_screen_ctx(chat_id),
        variables={
            "events_list": events_md,
            "has_events": "1" if has else "",
            "no_events": "" if has else "1",
        },
        t_func=_t_by_lang,
    )
    render_screen(screen, chat_id, bot)
```

**When per-event buttons are needed** (e.g., each event has its own "Open" button), the handler builds `ButtonRow` widgets from `bot_ui.py` directly and appends them to the rendered `screen.widgets` list before passing to `render_screen`:

```python
# Inject dynamic event buttons after loading the static layout
for ev in events[:10]:
    screen.widgets.insert(-2, ButtonRow(buttons=[  # before the last 2 rows (Add/Console, Back)
        Button(label=f"📅 {ev['title']}", action=f"cal_event:{ev['id']}"),
    ]))
```

This "YAML skeleton + Python injection" approach keeps the static layout in YAML while allowing the dynamic parts to be built from data.

### 3.3 Pattern C — Wizard Step Template (Reusable)

For sequential wizards (Contact add, Mail setup), a single YAML template can serve all steps:

```yaml
# wizard_step.yaml — generic step template
title: "{step_title}"
parse_mode: Markdown

widgets:
  - type: markdown
    text: "{step_body}"
    visible_if: has_body

  - type: text_input
    placeholder: "{placeholder}"
    action: "{submit_action}"

  - type: button_row
    visible_if: has_skip
    buttons:
      - label_key: contact_btn_skip
        action: "{skip_action}"
      - label_key: btn_cancel
        action: cancel

  - type: button_row
    visible_if: no_skip
    buttons:
      - label_key: btn_cancel
        action: cancel
```

The Python handler selects variables per step:

```python
def _show_wizard_step(chat_id, step_name, ...):
    screen = load_screen("screens/wizard_step.yaml",
        user=_screen_ctx(chat_id),
        variables={
            "step_title": _t(chat_id, f"contact_add_prompt_{step_name}"),
            "step_body": hint_text or "",
            "has_body": "1" if hint_text else "",
            "placeholder": _t(chat_id, f"contact_placeholder_{step_name}"),
            "submit_action": f"contact_step_submit:{step_name}",
            "skip_action": "contact_add_skip",
            "has_skip": "1" if step_name != "name" else "",
            "no_skip": "" if step_name != "name" else "1",
        },
        t_func=_t_by_lang,
    )
    render_screen(screen, chat_id, bot)
```

## 4. Implementation Roadmap

### Phase 1 — Calendar Screens (Highest Impact)

Calendar has the most user-facing screens. Convert these 5:

| YAML Screen | Replaces | Variables |
|---|---|---|
| `cal_add_prompt.yaml` | `_start_cal_add()` inline text | — |
| `cal_confirm.yaml` | `_show_cal_confirm()` | `{event_title}`, `{event_dt}`, `{remind_text}` |
| `cal_confirm_multi.yaml` | `_show_cal_confirm_multi()` | same + `{step_n}`, `{step_total}` |
| `cal_event_detail.yaml` | `_handle_cal_event_detail()` | `{ev_id}`, `{title}`, `{dt}`, `{remind}`, `{countdown}` |
| `cal_delete_confirm.yaml` | `_handle_cal_delete_request()` | `{ev_id}`, `{title}`, `{dt}` |

**Effort:** ~4 hours. Flow logic stays in Python unchanged. Only the keyboard/text construction moves to YAML.

### Phase 2 — Contacts Screens

| YAML Screen | Replaces | Variables |
|---|---|---|
| `contacts_menu.yaml` | `_handle_contacts_menu()` | — |
| `contact_detail.yaml` | `_handle_contact_open()` | `{name}`, `{phone}`, `{email}`, `{notes}`, `{slug}` |
| `contact_delete_confirm.yaml` | `_handle_contact_delete()` | `{slug}`, `{name}` |

**Effort:** ~2 hours.

### Phase 3 — Mail Screens

| YAML Screen | Replaces |
|---|---|
| `mail_consent.yaml` | `handle_mail_consent()` |
| `mail_provider.yaml` | `handle_mail_consent_agree()` |
| `mail_settings.yaml` | `handle_mail_settings()` |
| `mail_digest.yaml` | `handle_digest_auth()` |

**Effort:** ~2 hours.

### Phase 4 — Error Protocol & Documents

| YAML Screen | Replaces |
|---|---|
| `errp_name_prompt.yaml` | `_start_error_protocol()` |
| `errp_collect_status.yaml` | Collection status display |
| `docs_menu.yaml` | `_handle_docs_menu()` |

**Effort:** ~1 hour.

### Phase 5 — Admin Sub-Screens (Optional)

Admin screens (LLM menu, voice opts, user management) are admin-only and change rarely. Converting them provides less user-facing value but improves consistency.

**Effort:** ~4 hours (many screens, each small).

## 5. What Stays in Python (Not Convertible)

These operations **must remain in Python** — they involve runtime logic, async I/O, or Telegram API specifics:

| Operation | Why |
|---|---|
| LLM calls (`ask_llm()`, JSON parsing) | Async, error handling, timeout |
| IMAP test (`imaplib.IMAP4_SSL`) | Network I/O, credential handling |
| File I/O (note read/write, calendar JSON) | Data layer |
| `ForceReply` prompts | Telegram-specific message type |
| Voice message download + STT pipeline | Binary I/O + subprocess |
| `threading.Timer` for reminders | Runtime scheduling |
| `_user_mode` state transitions | Flow control |

**Rule of thumb:** If it involves I/O, LLM, or flow control → Python. If it involves text, layout, or buttons → YAML.

## 6. DSL Enhancement: `url` Property for Buttons (Optional)

Some buttons need an external URL instead of a callback action (e.g., Web link). The schema already supports `url` on `button_object` but only `action` is required. The `_build_button` function should check for `url` and pass it through:

```python
@_register("button")
def _build_button(w: dict, **ctx) -> Button:
    label = _resolve_text(w, "label_key", "label", ...)
    action = _resolve_action(w, ctx["vars"])
    url = _substitute(w.get("url", ""), ctx["vars"]) or None
    return Button(label=label, action=action, url=url, style=w.get("style", "primary"))
```

This enables YAML buttons with external links without any structural DSL change.

---

## 7. Summary & Recommendation

**Recommended approach:** Pattern A (YAML Layout Screens) + Pattern B (Dynamic List Injection) for all new screen conversions. This requires **zero DSL changes** and achieves 80% of the benefit.

**Not recommended:** Building a YAML-based workflow engine or state machine. The complexity-to-benefit ratio is poor, and Python handles flow control better.

| Metric | Current state | After Phase 1–4 |
|---|---|---|
| Total YAML screens | 10 | ~25 |
| % of user-facing layouts in YAML | ~30% | ~75% |
| Lines saved in Python | — | ~400–600 lines (keyboard builders + inline text) |
| New DSL features needed | — | 0 |
| Risk | — | Low (layout-only changes, logic unchanged) |
