# Pico Gen 1.0 — Multimodal UX Concept

**Date:** March 2026  
**Status:** Design Concept — not yet implemented  
**Companion:** `doc/web-ui/mockups-gen1/index.html` — interactive mockup (8 screens)

---

## 1. Problem Statement

The current Pico web interface is **navigation-first**: the user must know which page contains the action they want, navigate there, then fill in a form. This is a technical mindset, not a conversational one.

| Current (navigation-first) | Gen 1.0 (intent-first) |
|---|---|
| User taps "Notes → New Note" | User says "Create a note about..." |
| User navigates to "Calendar" | User says "Add a meeting tomorrow at 3" |
| User remembers which page has what | User expresses what they need |
| Voice is a separate page (`/voice`) | Voice is available everywhere, always |
| "Modes" are visual pages | "Modes" are transparent — the UI navigates itself |

**Core insight:** A truly helpful assistant should allow the user to forget the interface exists. The goal is zero navigation friction.

---

## 2. Core Principles

### P1 — Voice First, Text Equal
Voice is the primary input method. Typing is fully supported — but the design defaults to voice. The mic orb is always present. One tap away from any state.

### P2 — Intent Over Navigation
No "go to Notes then click New". The user expresses intent from anywhere. The system routes to the right action.

### P3 — Transparent Understanding
Before executing, the system *shows* what it understood — the parsed intent + extracted entities. The user sees a correction window (2 seconds) before the action runs. This eliminates fear of misrecognition.

### P4 — Confirmation Without Friction
Destructive or complex actions get a confirmation card. Read-only actions (show me my notes, what's the weather) execute immediately. The rule: if it can't be easily undone, confirm. If it's read-only, proceed instantly.

### P5 — Contextual Continuity
After every action, the assistant offers 2–3 natural next steps — generated by the LLM based on what was just completed. The conversation never goes cold.

### P6 — One Thread, All Modes
All interactions appear in a persistent conversation thread. Switching from notes to chat to calendar is not page navigation — it's the thread continuing with different card types.

## 3. The Five Universal States

Every interaction — whether voice or text, simple or complex — passes through exactly five states:

```
IDLE → LISTENING → UNDERSTANDING → ACTING → COMPLETING
```

| State | Visual cue | What the user experiences |
|---|---|---|
| **Idle** | Orb breathes gently (scale ↔ 1.07) | "I'm here when you need me" |
| **Listening** | Orb expands, waveform visible, live transcript | "I can hear you" |
| **Understanding** | Orb shrinks, Intent Card slides up, shows what was parsed | "I see what you mean" |
| **Acting** | Relevant module opens, data streams in, voice continues if needed | "I'm doing it" |
| **Completing** | Success orb (green), assistant message, What's Next chips | "Done. What would you like next?" |

## 4. Key Flows

### Flow A — Create Note via Voice (full flow, as in the mockup)

```
[Hub — Idle]
  → User taps orb OR speaks

[Listening]
  🎤 "Create a new note called My Cookbook for Tomorrow
       with a table of contents based on the content"

[Intent Card appears — 2s window]
  📝 Creating a Note
  Title: "My Cookbook for Tomorrow"
  TOC: Auto-generate ✓
  Mode: Voice dictation
  [✓ Proceed] [✎ Correct title] [✗]

[Note Editor opens — title pre-filled]
  🎤 Listening badge active in topbar
  "Now dictate the content — I'll listen."
  
  User dictates: "Section one, Soup. Ingredients: carrots, celery,
  vegetable broth. Section two, Bread. Simple sourdough recipe..."
  
  → Content streams into editor
  → TOC auto-generates: 1. Introduction · 2. Soup · 3. Bread

[Confirmation Card]
  📝 Save this note?
  Preview: title, section count, word count, TOC snippet
  [✅ Save] [✎ Edit] [✗ Cancel]

[Success + What's Next]
  ✅ Note saved!
  "My Cookbook for Tomorrow · 3 sections · ~180 words"
  
  Assistant: "Your note is saved! Want me to read it back to you?"
  
  [💬 Chat about it] [🔊 Read aloud] [📝 New note]
  [✉ Share by email] [🗓 Add cooking event]

User taps "💬 Chat about it"
→ Chat mode activates, note card appears in thread
→ Assistant has context of the note
```

```
[Hub]
  🎤 "What notes do I have?"

→ No confirmation needed — read-only
→ Orb pulses briefly
```

### Flow D — Mode Switch via Voice

```
[Success state after any action]
  User: 🎤 "I want to chat with you"

→ Intent: SWITCH_MODE · target: chat
→ No confirmation (instant, reversible)
→ Chat thread opens
→ Assistant: "Chat mode! What would you like to talk about?"
→ Previous action card visible in thread for context
```

### Flow E — Uncertain Recognition + Correction

```
[Listening]
  🎤 "Meeting with Anna... no wait, with the design team
       on Friday at... hmm, at ten in the morning"

[Intent Card — with uncertainty marker]
  🗓 Adding Calendar Event
  Title: "Meeting with [design team?]"  ← uncertain highlight
  Date: Friday · 10:00
  [✓ Proceed] [✎ Correct title] [✗]

User taps "✎ Correct title"
  → Title field highlights
  🎤 "with the product team"
  → Title updates: "Meeting with the product team"
  → Re-shows intent card with corrected value
  → User taps ✓ Proceed
```

## 5. The Intent Card Component

The critical innovation. Appears before every non-trivial action.

```
╔══════════════════════════════════════════════════╗
║  📝  Creating a Note              97% confident  ║
║  ────────────────────────────────────────────    ║
║  Title    "My Cookbook for Tomorrow"             ║
║  TOC      Auto-generate ✓                        ║
║  Mode     Voice dictation                        ║
║                                                  ║
║  [✓ Proceed now]   [✎ Correct]   [✗ Cancel]     ║
║  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░░ proceeding in 1s   ║
╚══════════════════════════════════════════════════╝
```

**Rules:**
- Always show before modifying/creating data
- Never show for read-only queries
- Auto-proceeds after 2 seconds — no button required for confident users
- Confidence < 75%: no auto-proceed, require explicit confirmation
- Uncertain entities shown with `[?]` marker

## 6. The Voice Orb

The single persistent interactive element. Lives bottom-right on desktop, center-bottom on mobile.

| State | Appearance | Behavior |
|---|---|---|
| Idle | 60px, purple gradient, breathing 0.95↔1.07 | Tap → start listening |
| Hover | Glow intensifies | Tooltip: "Tap to speak" |
| Listening | 80px, waveform rings, bright glow | Tap → stop, or auto-stop on silence |
| Processing | 52px, spinning ring, shrinks | Non-interactive |
| Success | Green gradient, checkmark, pulse | Returns to idle after 1.5s |
| Error | Red pulse, returns to idle | Error message in thread |

---

## 7. What's Next Rail

Appears after every completed action. 2–5 context-generated chips. Auto-dismisses after 15 seconds.

**Generation rule:** LLM receives `{action, result, user_history}` → returns 3–5 next-step suggestions as JSON. Examples:

| After | Suggested chips |
|---|---|
| Note saved | 💬 Chat about it · 🔊 Read aloud · 📝 New note · ✉ Share |
| Event added | 🗓 See all events · ➕ Add another · 📧 Send invite |
| Mail read | 💬 Help me reply · 📝 Note key points · 🗓 Add deadline |
| Voice command | 📝 Save transcript · 💬 Continue chatting |

---

## 8. Thread Card Types

The unified thread replaces page-based navigation. Everything that happens creates a card in the thread.

| Type | Example content | Quick actions |
|---|---|---|
| Note Card | 📝 Title · word count · excerpt | Open · Edit · Read aloud |
| Calendar Card | 🗓 Event · date · countdown | Edit · Delete · Share |
| Chat Message | 💬 User/bot bubble (text or voice) | Copy · TTS · React |
| Mail Card | 📧 Subject · sender · summary | Reply · Forward · Note |
| Status Card | ⚙️ System event, version, error | Dismiss |

## 9. Telegram Adaptation

The same intent model maps directly to Telegram:

| Web component | Telegram equivalent |
|---|---|
| Voice Orb tap | Send a Telegram voice message |
| Intent Card | Bot text + 2-button inline keyboard |
| Confirmation Card | Bot card + [✅ Yes] [✏️ Edit] [❌ Cancel] |
| What's Next chips | Inline keyboard row after action complete |
| Thread Note Card | Text message + [📖 Open] [✏️ Edit] [🔊 TTS] buttons |
| "Open in web" | Deep-link button to the specific item in web UI |

**Example (note creation in Telegram):**
```
User:  🎤 voice message (6s)

Bot:   📝 I understood: Create Note
       Title: "My Cookbook for Tomorrow"
       [✅ OK]  [✎ Correct title]

User:  ✅ OK

Bot:   Title set! Now send me the content (voice or text).

User:  🎤 voice message (18s) — dictates sections

Bot:   ✅ Ready to save:
       "My Cookbook for Tomorrow" · 3 sections · ~180 words
       [✅ Save note]  [✎ Edit]  [✗ Cancel]

User:  ✅ Save note

Bot:   ✅ Note saved!
       [💬 Chat about it] [📝 New note] [🔊 Read aloud] [✉ Share]
```

## 10. Architecture: One Intent API, Two Renderers

```
User input (voice or text)
        │
        ▼
    /api/intent  ──────────────────────────────────
     LLM parser                                    │
     returns: {intent, entities, confidence}       │
        │                                          │
        ├── Web Renderer                           │
        │   Intent Card → Action Module → Thread  │
        │                                          │
        └── Telegram Renderer                      │
            Text card + inline keyboard            │
                                                  ─┘
```

Adding WhatsApp, Discord, or any 3rd channel requires only a new renderer — the intent logic is shared.

**New backend endpoint needed:**
```
POST /api/intent
  body: { text: string, audio_b64?: string }
  returns: {
    intent: "create_note" | "add_event" | "query" | "switch_mode" | ...,
    entities: { title?, dt_iso?, content?, target_mode? },
    confidence: 0.0–1.0,
    auto_proceed: boolean
  }
```

## 11. Implementation Phases

### Phase M0 — Foundation (1–2 days)
- [ ] `/api/intent` endpoint in `bot_web.py`
- [ ] Voice Orb JS component (reusable, CSS + vanilla JS)
- [ ] Intent Card component

### Phase M1 — Voice Hub (2–3 days)  
- [ ] Hub page (`/hub`) with orb + recent strip + quick chips
- [ ] MediaRecorder → `/api/intent` → Intent Card → route to action
- [ ] `base.html`: Hub as first sidebar item

### Phase M2 — Unified Thread (3–4 days)
- [ ] Thread model in session (dict, or SQLite for persistence)
- [ ] Thread card renderers for all content types
- [ ] What's Next rail (LLM-generated, auto-dismiss)

### Phase M3 — Full Voice Flows (2–3 days)
- [ ] End-to-end: voice → note creation with inline dictation
- [ ] End-to-end: voice → calendar add
- [ ] Mode switching via voice command

### Phase M4 — Polish + Telegram Sync (2–3 days)
- [ ] Telegram: intent-based inline keyboard generation
- [ ] Deep links (Telegram → Web for rich editing)
- [ ] Error recovery, uncertainty correction flow

## 12. Open Questions

1. **Session persistence**: Should the thread persist across browser sessions? (SQLite vs session cookie)
2. **Intent ambiguity**: If confidence < 50%, show a clarification dialog rather than an intent card?
3. **Orb position**: Bottom-right (always accessible) vs bottom-center (prominent, mobile-friendly)?
4. **TOC generation**: Auto-detect section headers in voice dictation using NLP keywords ("section one", "next"), or prompt LLM to structure the content?
5. **Thread card limit**: Cap at N cards per session to avoid unbounded growth? Rolling window?
